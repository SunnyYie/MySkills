# Bugfix Orchestrator Agent + Skills + CLI 技术方案

## 1. 文档目标

本文档基于《需求文档》给出 v1 技术实现方案，面向本地单用户、CLI-first 的 bug 修复编排场景，定义系统分层、核心模块、关键数据模型、状态机、持久化、CLI 设计、外部集成与安全策略。

本文档目标是把产品需求收敛成可实现的工程设计，作为后续详细设计、任务拆解和实现评审的依据。本文档不包含具体代码实现。

## 2. 设计范围与边界

### 2.1 v1 覆盖范围

v1 覆盖以下最小闭环：

- 从 Jira issue 拉取 bug 信息
- 解析项目、需求、仓库和模块上下文
- 生成 `Requirement Brief`
- 生成代码定位建议与修复计划
- 支持审批、回退、恢复执行
- 支持 GitLab 产物与验证结果补录
- 支持 Jira 回写预览与执行
- 支持飞书文档写入预览与执行
- 导出统一 `Bugfix Report`

### 2.2 v1 非范围

v1 不包含：

- 自动修改代码
- 自动执行测试
- 自动创建 commit、branch、MR
- 服务端多用户调度
- 自然语言会话式入口
- 复杂凭证托管平台
- 高自治知识推理

## 3. 总体架构

系统采用五层架构：

```text
CLI Layer
  -> Workflow/Agent Layer
    -> Skill Layer
      -> Infrastructure Layer
        -> External Systems / Local Repo
```

其中 `Infrastructure Layer` 统一承接外部系统与本地仓库访问，避免 workflow 或 skill 越层直接读写底层资源。

### 3.1 分层职责

#### CLI 层

负责：

- 命令解析
- 参数校验
- 交互式确认
- 结果展示
- JSON/TTY 输出切换
- dry-run 开关透传
- 错误码返回

CLI 不持有业务运行逻辑，不直接修改业务状态，只负责把用户输入转成 workflow 命令并展示结果。

#### Workflow/Agent 层

负责：

- 按固定阶段编排主流程
- 维护 `ExecutionContext`
- 驱动状态流转
- 维护审批状态与审批历史
- 决定何时生成 preview、何时允许真实写入
- 写入 checkpoint
- 实现恢复执行
- 记录审计与副作用账本

Workflow/Agent 层是唯一业务状态持有者，也是审批与回退逻辑的唯一 owner。

#### Skill 层

负责：

- 封装结构化能力
- 接受输入上下文并返回结构化结果
- 保持无状态、可复用、可独立测试

Skill 只做“读取输入 -> 计算结果”的纯逻辑，不记录审批结果、不持久化 run state、不直接操作 checkpoint、不负责输出渲染。

#### Infrastructure 层

负责：

- 对接 Jira、GitLab、飞书
- 对接本地代码仓库与 Git 元数据
- 目标解析
- 能力检查
- 外部读取
- 写入预览
- 真实写入
- 外部系统错误标准化

Infrastructure 层是唯一外部系统与本地仓库访问入口。

#### Renderers

负责：

- CLI 展示渲染
- Markdown 导出
- JSON 输出格式化

渲染职责只在 renderer 中实现，不再由 skill 重复承担。

## 4. 关键设计原则

- 显式配置优先，禁止高风险静默推断
- 所有关键节点允许人工确认
- 所有外部写操作默认先 preview，再 execute
- 每阶段必须产出结构化结果
- 运行过程必须可恢复、可审计、可脱敏
- Skill 之间通过上下文编排，不直接互相依赖
- Workflow 是唯一状态源
- Infrastructure 是唯一外部系统与本地仓库读写入口
- 批准不等于执行副作用，真实写入必须显式触发

## 5. 技术栈建议

v1 建议采用：

- 语言：TypeScript
- 运行时：Node.js
- CLI 框架：`commander`
- schema 校验：`zod`
- 测试框架：`vitest`
- 文件系统操作：Node.js `fs/promises`
- Markdown 渲染：内部模板拼装

选择原因：

- 适合构建本地 CLI 工具
- 类型系统适合约束配置、状态机和结构化契约
- 生态成熟，便于后续扩展 MCP、更多 connector 和 renderers

## 7. 核心数据模型

### 7.1 ProjectProfile

`ProjectProfile` 是项目画像配置，是运行期唯一可信配置来源。

最小结构包含：

- `project_id`
- `project_name`
- `config_version`
- `jira`
- `requirements`
- `gitlab`
- `feishu`
- `repo`
- `approval_policy`
- `serialization_policy`
- `sensitivity_policy`

设计要求：

- 按项目独立存储
- 可版本化
- 可校验完整性
- 支持凭证引用隔离
- 支持 requirement link rules 优先级与回退策略

### 7.2 ExecutionContext

`ExecutionContext` 是唯一业务运行态对象，由 workflow 持有。

最小字段包含：

- `run_id`
- `project_id`
- `config_version`
- `run_mode`
- `run_lifecycle_status`
- `run_outcome_status`
- `current_stage`
- `stage_status_map`
- `stage_artifact_refs`
- `active_approval_ref_map`
- `waiting_reason`
- `initiator`
- `started_at`
- `updated_at`
- `jira_issue_snapshot_ref`
- `requirement_refs`
- `repo_selection`
- `code_targets`
- `root_cause_hypotheses`
- `fix_plan`
- `verification_plan`
- `verification_results_ref`
- `gitlab_artifacts`
- `jira_writeback_draft_ref`
- `jira_writeback_result_ref`
- `feishu_record_draft_ref`
- `feishu_record_result_ref`
- `active_error_ref`
- `sensitive_field_paths`

设计约束：

- `ExecutionContext` 只保存“当前业务有效态”
- `context.json` 只允许保存索引型字段、摘要字段、hash 和 artifact ref
- 自由文本、长备注、外部正文、补录原文不得直接内嵌在 `ExecutionContext`
- 审计事件、副作用账本、checkpoint 元数据不直接内嵌在 context 中
- 过期产物通过 `stage_artifact_refs` 与 artifacts 历史引用管理
- 支持完整序列化
- 支持恢复执行
- 支持 CLI 摘要展示

## 8. 状态机设计

系统必须严格区分三类状态，并定义状态不变式。

### 8.1 阶段状态

- `not_started`
- `in_progress`
- `output_ready`
- `waiting_approval`
- `approved_pending_write`
- `executing_side_effect`
- `waiting_external_input`
- `completed`
- `failed`
- `stale`
- `skipped`

说明：

- `approved_pending_write` 表示 preview 已批准，但真实写入尚未执行
- `executing_side_effect` 表示正在执行 Jira/飞书写入
- `skipped` 用于子工作流中不适用的阶段

### 8.2 审批决定

- `approve`
- `reject`
- `revise`

### 8.3 审批状态

- `none`
- `pending`
- `approved`
- `rejected`
- `revise_requested`
- `superseded`

### 8.4 run 状态

#### `run_lifecycle_status`

- `active`
- `waiting_approval`
- `waiting_external_input`
- `paused`
- `cancelled`
- `completed`
- `failed`

#### `run_outcome_status`

- `unknown`
- `in_progress`
- `success`
- `partial_success`
- `failed`
- `cancelled`

### 8.5 状态不变式

- `Workflow` 是状态唯一写入者
- 任一阶段为 `waiting_approval` 时，`run_lifecycle_status` 必须为 `waiting_approval`
- 任一阶段为 `waiting_external_input` 时，`run_lifecycle_status` 必须为 `waiting_external_input`
- 阶段只有在最终结果落盘后才能进入 `completed`
- 对 Jira/飞书写入阶段，`approve` 只表示通过审批，不等于执行写入
- `run_outcome_status` 不表达等待态，只表达当前累计结果
- `partial_success` 只在存在已成功结果且存在不可忽略失败时出现

### 8.6 状态流转规则

- 阶段开始执行时进入 `in_progress`
- 阶段结构化输出生成后进入 `output_ready`
- 有审批门的阶段由 `output_ready` 转入 `waiting_approval`
- 审批 `approve` 后：
  - 对纯分析阶段，进入 `completed`
  - 对含外部写入的阶段，进入 `approved_pending_write`
- 审批 `reject` 后，`run_lifecycle_status` 进入 `cancelled`
- 审批 `revise` 后，回退到指定上游阶段
- 进入真实写入时，阶段进入 `executing_side_effect`
- 写入成功后阶段进入 `completed`
- 写入失败后阶段进入 `failed` 或保持 `approved_pending_write`
- 当阶段等待用户补录 GitLab 产物或验证结果时，进入 `waiting_external_input`

### 8.7 revise 与 stale 规则

当用户选择 `revise` 时：

- `rollback_to_stage` 的当前有效 artifact/ref 与当前审批一并失效
- `rollback_to_stage` 进入 `not_started`
- 其后所有阶段状态置为 `stale`
- 其后所有审批记录状态置为 `superseded`
- 其后所有 draft/result 引用从当前有效态移除，但历史 artifacts 保留
- 已经成功执行的外部写入不可回滚，只保留为历史事实

恢复执行时只能读取“当前有效版本”产物，不能读取 `stale` 产物。

### 8.8 检查点策略

每次 durable 状态迁移都必须落盘 checkpoint，至少包括：

- 阶段开始
- 阶段输出生成
- 进入等待审批
- 审批结果写入
- 进入等待外部输入
- 接收外部补录输入
- 进入真实写入
- 外部写成功
- 外部写失败
- 回退生效
- 恢复重绑定当前阶段

checkpoint 必须能独立回答以下问题：

- 当前恢复点对应的业务版本是什么
- 当前有效 preview / artifact / approval 是哪些
- 最近一次副作用尝试是否已进入 `prepared` 或 `dispatched`
- 指定 `--checkpoint <id>` 恢复后应从哪一步继续

恢复执行要求：

- 基于最近持久化 checkpoint 恢复
- 已审批通过阶段默认不静默重跑
- 已成功外部写入必须通过副作用账本去重
- `waiting_external_input` 恢复后继续等待输入，不重放副作用
- 若错误为 `outcome_unknown = true`，恢复时必须先对账，再决定是否重试
- 若存在 `prepared` 或 `dispatched` 且无终态的副作用记录，恢复时必须优先进入 reconcile，而不是直接重发

## 9. 工作流设计

主流程固定为八个阶段：

1. `Intake`
2. `Context Resolution`
3. `Requirement Synthesis`
4. `Code Localization`
5. `Fix Planning`
6. `Execution`
7. `Artifact Linking`
8. `Knowledge Recording`

### 9.1 主流程说明

#### Intake

输入 Jira issue key，读取 bug 基础信息、描述、状态、标签和需求线索。

#### Context Resolution

根据项目画像解析：

- 所属项目
- 本地仓库
- 需求来源
- 模块候选
- requirement 绑定结果

#### Requirement Synthesis

生成 `Requirement Brief`，这是首个关键审批门。

#### Code Localization

基于上下文输出候选代码位置、影响模块和根因假设。

#### Fix Planning

生成修复建议、影响范围、验证建议，是第二个关键审批门。

#### Execution

v1 中不自动改代码，只负责：

- 挂起等待外部修复动作
- 接收补录验证结果
- 接收 GitLab 产物
- 标准化外部输入

#### Artifact Linking

该阶段拆为两个子步骤：

- 生成 Jira 回写 preview 并等待审批
- 审批通过后执行真实 Jira 写回

#### Knowledge Recording

该阶段拆为两个子步骤：

- 生成飞书写入 preview 并等待审批
- 审批通过后执行真实飞书写入

### 9.2 子工作流

除主流程外，支持三个子工作流：

- 仅生成 `Requirement Brief`
- 仅回写 Jira
- 仅写入飞书记录

设计要求：

- 子工作流复用同一套状态机规则
- 不适用阶段标记为 `skipped`
- 子工作流具备独立最小输入校验
- 不允许绕过必要审批

## 10. Skill 设计

### 10.1 统一 Skill 接口

每个 skill 统一遵循以下设计：

- 输入：显式 schema
- 输出：结构化 `StageResult<T>`
- 错误：统一 `StructuredError`
- 特性：无状态、可组合、可独立测试

调用规则：

- Workflow 先通过 storage / infrastructure 获取配置、issue 快照、repo 快照或外部输入
- Skill 只消费这些已获取的结构化对象并产出结构化结果
- Skill 不直接发起 I/O

`StageResult<T>` 建议包含：

- `status`
- `summary`
- `data`
- `warnings`
- `errors`
- `source_refs`
- `generated_at`

### 10.2 各 Skill 职责

#### `config-loader`

- 校验并标准化已读取的 `ProjectProfile`
- 输出标准化项目配置对象

#### `jira-intake`

- 解析已读取的 Jira issue snapshot
- 识别 issue 元数据和需求线索

#### `project-context`

- 根据项目画像将 bug 与需求、仓库、模块建立关联

#### `requirement-summarizer`

- 生成 `Requirement Brief`

#### `code-locator`

- 输出代码位置候选
- 输出影响模块
- 输出根因假设

#### `fix-planner`

- 输出修复建议
- 输出验证建议
- 输出开放风险

#### `verification-recorder`

- 校验并标准化外部补录验证结果
- 生成可纳入 `ExecutionContext` 的验证摘要

#### `gitlab-linker`

- 校验和标准化 GitLab 产物引用
- 生成 Jira 回写业务草稿

#### `feishu-recorder`

- 基于模板和上下文生成飞书记录业务草稿

#### `report-writer`

- 基于最终上下文输出 `Bugfix Report`

## 11. Infrastructure 设计

### 11.1 能力型接口设计

不再对所有 connector 强行使用一套大而全接口，而采用 capability-based 设计：

- `HealthCheckCapability`
- `ReaderCapability`
- `TargetResolverCapability`
- `PreviewWriterCapability`
- `SideEffectExecutorCapability`
- `ArtifactResolverCapability`
- `RepoWorkspaceCapability`

不同基础设施按能力组合实现，避免 GitLab 出现空的写入接口。

### 11.1.1 owner 约束

- Skill 只生成 canonical draft / structured content
- Infrastructure 把 draft 转成目标系统 preview / payload，并执行真实写入
- Renderers 只把 preview 或结果对象格式化成 CLI / Markdown / JSON

文档中“生成 preview”的 owner 统一归 Infrastructure，不再由 skill 承担。

### 11.2 Jira Connector

负责：

- 根据 issue key 获取 bug 信息
- 读取状态、描述、标签、关联线索
- 解析回写目标
- 生成 Jira 回写 preview
- 执行字段或评论写入
- 返回结果对象与目标引用

写入契约建议拆为三段：

#### Preview Input

- `issue_key`
- `target_type`
- `target_field_id_or_comment_mode`
- `resolved_target_snapshot`
- `content_source`

#### Execute Input

- `preview_ref`
- `rendered_preview`
- `request_payload_hash`
- `idempotency_key`
- `dedupe_scope`
- `expected_target_version`
- `conflict_policy`

#### Execute Result

- `result_id`
- `target_ref`
- `target_version`
- `result_url`
- `already_applied`
- `external_request_id`
- `updated_at`

补充要求：

- Jira comment 写入必须包含可外部观测的稳定幂等标记
- `execute-write` 前先按 marker 或等价规则查重
- `outcome_unknown` 恢复时优先按 marker 对账

### 11.3 GitLab Connector

负责：

- 标准化 commit/branch/MR 产物引用
- 输出标准链接
- 辅助 repo 默认分支和 project 信息解析

GitLab 在 v1 中主要实现 `ArtifactResolverCapability`，不承担 preview-write / execute-write。

产物引用至少包含：

- `artifact_source`
- `project_id`
- `project_path`
- `default_branch`
- `branch_name`
- `commit_sha`
- `commit_url`
- `mr_iid`
- `mr_url`
- `artifact_type`
- `created_at`

校验要求：

- `artifact_type = commit` 时，`commit_sha` 和 `commit_url` 必填
- `artifact_type = branch` 时，`branch_name` 必填
- `artifact_type = mr` 时，`mr_iid` 和 `mr_url` 必填
- `artifact_source` 必须区分系统内生成与外部导入

### 11.4 飞书 Connector

负责：

- 根据模板生成写入 preview
- 将记录写入指定文档锚点或块位置
- 支持独立补写

写入契约建议拆为三段：

#### Preview Input

- `space_id`
- `doc_id`
- `block_id_or_anchor`
- `template_id`
- `template_version`
- `write_mode`
- `resolved_target_snapshot`

#### Execute Input

- `preview_ref`
- `rendered_preview`
- `request_payload_hash`
- `idempotency_key`
- `dedupe_scope`
- `expected_target_version`
- `conflict_policy`

#### Execute Result

- `result_id`
- `target_ref`
- `target_version`
- `result_url`
- `already_applied`
- `external_request_id`

v1 默认 `write_mode = append`，同时保留 `replace_block` 扩展位。

对于 `append` 写入，必须定义稳定幂等标记，避免恢复执行时重复追加。

飞书 `append` 与 Jira `comment` 统一要求：

- preview 生成时注入稳定 marker
- 执行前查 marker
- 恢复时先对账 marker，再决定是否重试

### 11.5 Repo Workspace Adapter

负责：

- 根据 `repo.local_path` 打开本地仓库
- 读取 Git branch / commit / remote 信息
- 根据 `repo.module_rules` 解析模块候选
- 执行文件搜索、路径归一化、代码候选定位

该模块不直接改代码，只提供只读能力。

## 12. 持久化与审计设计

v1 采用本地文件系统持久化，不引入数据库。

### 12.1 存储路径建议

- 项目画像：`~/.config/bugfix-orchestrator/projects/<project_id>.json`
- run 数据：`~/.local/share/bugfix-orchestrator/runs/<run_id>/`

### 12.2 run 目录结构建议

```text
<run_id>/
  context.json
  events.ndjson
  side-effects.ndjson
  checkpoints/
  artifacts/
  lock
```

说明：

- `context.json` 保存最新业务有效态
- `context.json` 仅保存 allowlist 内的索引型字段、摘要与引用
- `events.ndjson` 保存审计事件
- `side-effects.ndjson` 保存副作用账本
- `checkpoints/` 保存阶段快照
- `artifacts/` 保存 brief、preview、report 等导出工件
- `lock` 用于 run 级互斥

### 12.3 原子写与文件锁

本地文件持久化必须满足：

- run 级文件锁，防止两个终端并发修改同一 run
- 使用 `tmp + fsync + rename` 原子落盘
- 目录权限至少 `0700`
- 文件权限至少 `0600`

此外还必须支持：

- 针对外部目标的全局去重索引或 target 级锁
- `run` 与 `record` 共用同一幂等检查与锁定机制

推荐写入顺序：

1. 写 artifact 或 ledger 明细
2. 写 checkpoint
3. 更新 `context.json`
4. 追加审计事件

### 12.4 审计与副作用账本

#### 审计事件

建议记录：

- run 创建
- 阶段开始
- 阶段产出
- 审批请求
- 审批决定
- 外部写预览
- 外部写成功
- 外部写失败
- 恢复执行
- 回退重算

#### 副作用账本

每次 Jira/飞书写操作必须记录：

- `system`
- `operation`
- `idempotency_key`
- `dedupe_scope`
- `request_payload_hash`
- `target_ref`
- `expected_target_version`
- `result_ref`
- `status`
- `already_applied`
- `external_request_id`
- `executed_at`

执行协议要求：

1. 先写 `prepared` 副作用记录
2. 再写进入 `executing_side_effect` 的 checkpoint
3. 再发送外部请求
4. 请求发出后更新为 `dispatched`
5. 根据返回结果更新为 `succeeded`、`failed` 或 `outcome_unknown`

作用：

- 去重
- 恢复执行防重复写
- 冲突检测
- 审计追溯
- 崩溃恢复时的 reconcile

### 12.5 落盘分级策略

为避免持久化与脱敏冲突，三类文件采用不同等级：

- `context.json`
  只保存业务有效态索引、摘要、hash 与引用，不保存自由文本正文与原始 `request_payload`
- `checkpoints/`
  保存阶段快照，但只允许落盘脱敏版本
- `artifacts/`
  保存面向人工阅读的导出物，默认保存脱敏版

默认不落盘原始 `request_payload`，只保存 hash、目标引用和脱敏 preview。

## 13. CLI 设计

CLI 程序名建议为 `bugfix-orchestrator`，简称 `bfo`。

### 13.1 命令分组

#### `bind`

用于维护项目画像与绑定关系：

- `bind project`
- `bind jira`
- `bind gitlab`
- `bind feishu`
- `bind repo`
- `bind requirements`

#### `inspect`

用于体检和预览：

- `inspect config --project <id>`
- `inspect graph --project <id>`
- `inspect connectors --project <id>`
- `inspect run --run <run_id>`

#### `run`

用于流程执行：

- `run start --project <id> --issue <key>`
- `run brief --project <id> --issue <key>`
- `run resume --run <run_id>`
- `run status --run <run_id>`
- `run approve --run <run_id> --stage <stage> --preview-ref <ref>`
- `run revise --run <run_id> --rollback-to <stage>`
- `run reject --run <run_id> --stage <stage>`
- `run provide-artifact --run <run_id>`
- `run provide-verification --run <run_id>`
- `run preview-write --run <run_id> --stage <stage>`
- `run execute-write --run <run_id> --stage <stage> --preview-ref <ref> --confirm <preview-hash>`

#### `record`

用于独立补写的便捷入口：

- `record jira --project <id> --issue <key>`
- `record feishu --project <id>`

说明：

- `record` 不绕开 workflow
- `record` 命令会隐式创建一个最小 run，并返回 `run_id`
- 后续审批、status、preview-write、execute-write、checkpoint、ledger 全部仍走统一 `run` 生命周期

### 13.2 全局参数

统一支持：

- `--json`
- `--dry-run`
- `--non-interactive`
- `--output <path>`
- `--checkpoint <id>`

### 13.3 命令语义约束

- `approve` 只改变审批状态，不直接触发真实写入
- `approve` 必须绑定到不可变的 `preview_ref`
- `preview-write` 负责生成或刷新待执行 preview，并产出新的 `preview_ref`
- `execute-write` 负责真实外部写入，且必须校验当前有效 preview 与被批准 preview 一致
- `provide-artifact` 和 `provide-verification` 只用于补录外部输入，支持 `--file`、`--json` 或显式字段输入
- `provide-artifact` 默认追加到当前有效 artifact 集合，重复输入按幂等键去重
- `provide-verification` 默认替换当前有效 verification 摘要，原始正文作为 artifact 引用保存
- `status` 用于查看当前 run 状态、等待原因、允许动作和下一步建议动作
- `record` 只是创建最小 run 的快捷入口，不是旁路执行入口

### 13.4 非交互与 dry-run 约束

- `--non-interactive` 模式下，`execute-write` 必须显式携带 `--confirm <preview-hash>`，否则返回 `confirmation_required`
- `--non-interactive` 模式下不得弹出二次确认
- `--dry-run` 允许外部读取与 preview 生成，但不得触发真实外部写入
- `--dry-run` 默认不得写入副作用账本成功记录
- `--dry-run` 是否持久化临时 preview，必须在输出中显式标记

### 13.5 输出策略

- TTY 模式输出摘要、提示和 preview
- JSON 模式输出结构化对象
- 错误输出统一使用 `StructuredError`

`run status --json` 至少应输出：

- `run_id`
- `run_lifecycle_status`
- `run_outcome_status`
- `current_stage`
- `stage_status_map`
- `allowed_actions`
- `waiting_reason`
- `required_inputs`
- `active_preview_ref`
- `last_checkpoint_id`
- `next_step_examples`

### 13.6 错误码策略

建议至少区分：

- 配置错误
- 输入错误
- 认证失败
- 权限不足
- 外部写失败
- 外部写结果未知
- 审批拒绝
- 确认缺失
- 非法状态迁移
- 预览版本过期
- 锁冲突
- 版本冲突
- 流程等待外部输入
- 部分成功

## 14. 审批与人工确认设计

审批门固定设置在以下阶段后：

- `Requirement Synthesis`
- `Fix Planning`
- `Artifact Linking`
- `Knowledge Recording`

审批设计要求：

- 审批结果必须独立记录到审批历史
- `approve` 表示通过审批，不等于执行副作用
- `reject` 表示终止当前 run，但保留上下文
- `revise` 表示回退到指定上游阶段，并将后续产物标记为过期
- 当旧审批因回退失效时，必须标记为 `superseded`

为避免误操作，CLI 在真实写入前必须再次展示 preview 摘要。

若 preview 被刷新、回退或恢复导致版本变化，旧审批必须失效并要求重新审批。

## 15. 未绑定需求处理设计

系统必须支持 `unresolved` 需求绑定状态。

处理策略如下：

- 允许继续生成 `Requirement Brief`
- 允许继续生成代码定位建议与修复计划
- `Requirement Brief`、`ExecutionContext`、`BugfixReport` 必须显式保留 `binding_reason`
- 飞书记录允许写入，但“所属需求”必须标记为“未绑定需求”
- 当项目配置要求 requirement binding 为强约束时，必须在外部写回前阻塞

该策略保证：

- 不因映射失败完全阻断分析流程
- 同时保留强约束项目的治理能力

## 16. 安全与脱敏设计

### 16.1 凭证策略

v1 只支持：

- 环境变量引用
- 本地安全存储引用

`ProjectProfile` 中只存 `credential_ref`，不存明文 token。

### 16.2 落盘限制

以下信息不得以明文落盘：

- token
- cookie
- Authorization header
- 其他敏感认证材料
- 原始外部写入 `request_payload`

### 16.3 redaction 机制

`ExecutionContext` 与 `BugfixReport` 都必须支持 redaction。

通过 `sensitive_field_paths` 和落盘 allowlist 联合控制以下输出：

- 日志
- checkpoint
- report
- JSON 输出
- artifacts

### 16.4 日志策略

- 默认终端日志脱敏
- 审计事件只保存目标引用、payload hash、结果引用
- 不保存不必要的外部系统敏感正文
- 遇到 `outcome_unknown` 时必须记录外部 request id 或 trace id

## 17. 错误处理设计

统一覆盖以下错误类型：

- 配置缺失
- 认证失败
- 权限不足
- 网络错误
- 外部字段缺失
- 需求映射失败
- 仓库定位失败
- 用户拒绝执行
- 写回失败
- 写回结果未知

设计要求：

- 返回结构化错误
- 给出清晰用户提示
- 提供下一步建议动作
- 保留当前上下文
- 支持 `partial_success`
- 对 `outcome_unknown = true` 的场景禁止直接盲重试

部分成功示例：

- Jira 写回成功，飞书写入失败
- Brief 生成成功，但后续等待外部补录

结果未知示例：

- 外部请求超时，但服务端可能已成功写入

## 18. 非功能设计

### 18.1 易用性

- CLI 命令分组清晰
- 常见操作具备 preview 与提示
- 支持交互式和 JSON 两种输出模式

### 18.2 可恢复性

- durable 状态迁移级 checkpoint
- 基于文件系统的稳定持久化
- run 级文件锁
- 幂等账本去重副作用

### 18.3 可扩展性

- 后续可新增 connector
- 后续可接入 MCP
- 后续可新增 workflow 和 skill
- 当前分层可支持更多外部知识沉淀目标

### 18.4 可维护性

- 统一 schema
- 统一错误模型
- 能力型 infrastructure 接口
- 统一 renderer 接口
- workflow 与 skill 解耦

## 19. 方案总结

本方案将系统定义为一个以 `ExecutionContext` 为中心、以 workflow 为唯一状态 owner 的本地编排引擎：

- CLI 负责入口与展示
- Workflow 负责状态、审批、回退、恢复与审计
- Skill 负责无状态结构化能力
- Infrastructure 负责外部系统与本地仓库访问
- Renderers 负责统一输出

相比初版方案，本版进一步收敛了以下关键风险：

- 明确“审批不等于真实写入”
- 补齐 `approved_pending_write`、`executing_side_effect`、`skipped` 等关键状态
- 增加审批状态机与回退失效规则
- 引入本地仓库基础设施模块
- 让 connector 改为能力型接口，避免抽象过度
- 补齐 CLI 命令闭环
- 增加文件锁、原子写、落盘分级与 `outcome_unknown` 处理

该设计更适合作为后续详细设计、任务拆解和实现落地的基线版本。
