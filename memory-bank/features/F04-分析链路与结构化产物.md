# F04 分析链路与结构化产物

## 1. 目标与用户价值

本 feature 负责从只读输入生成结构化分析结果，包括 `Requirement Brief`、代码定位建议、修复计划、验证建议以及后续写回和报告所需的草稿输入。

用户价值如下：

- 用户可以在不改代码的前提下获得可审阅的分析结果。
- 所有分析能力都以结构化产物输出，便于审批、回退、恢复和最终报告复用。
- 分析链路与外部写回分离，便于保持 skill 无状态与可测试。

## 2. 来源依据

### 2.1 已确认事实

- Skill Layer 只做“结构化输入 -> 结构化输出”的纯逻辑。
- `Requirement Brief`、代码定位建议和修复计划是 v1 闭环中的核心分析产物。
- 所有 skill 输出统一使用 `StageResult<T>`。

### 2.2 上游约束

- 技术方案列出了 `jira-intake`、`project-context`、`requirement-summarizer`、`code-locator`、`fix-planner`、`verification-recorder`、`gitlab-linker`、`feishu-recorder`、`report-writer` 等 skill。
- 实施计划任务四冻结了 skill 清单、统一输出封装、需求映射分支、分析产物最小字段集、canonical draft 与错误语义。
- 审批逻辑、渲染逻辑和 connector 路由逻辑不作为 skill。

### 2.3 本 Feature 实现决策

- 以 `StageResult<T>` 作为唯一 skill 外壳。
- 将分析类产物和写入类 canonical draft 都视为结构化数据，不输出 CLI 或 Markdown 渲染结果。
- `gitlab-linker` 与 `feishu-recorder` 在本 feature 中只负责草稿输入，不负责 preview 正式生成。

## 3. 范围

### 3.1 本 Feature 要做什么

- 定义各 skill 的职责、输入输出和错误语义。
- 定义 `StageResult<T>` 字段和通用结果表达。
- 定义 `RequirementBrief`、代码定位结果、修复计划、验证建议、报告草稿输入的最小字段集。
- 定义需求映射的结构化分支结果。
- 定义 Jira / Feishu canonical draft 的输出基础。

### 3.2 本 Feature 明确不做什么

- 不落 checkpoint。
- 不维护 run state。
- 不生成 CLI 或 Markdown 呈现。
- 不生成 preview 或执行真实写入。

## 4. 前置依赖与输出产物

前置依赖：

- F01 提供配置与需求映射规则。
- F03 提供只读外部输入。

输出产物：

- 统一 `StageResult<T>`
- `RequirementBrief`
- 代码定位结果
- 修复计划与验证建议
- verification 结构化输入摘要
- Jira / Feishu canonical draft
- `BugfixReport` 的组装输入

## 5. 涉及的核心对象、状态与能力边界

核心对象：

- `RequirementBrief`
- `BugfixReport`
- `StageResult<T>`
- `StructuredError`

关键状态或字段：

- `requirement_binding_status`
- `binding_reason`
- `warnings`
- `errors`
- `source_refs`
- `generated_at`

能力边界：

- skill 不拥有审批状态。
- skill 不产生 preview 或外部 payload 真值。
- report-writer 负责汇总，但不决定主流程状态。

## 6. 实现设计

### 6.1 主要流程

主要流程如下：

1. `jira-intake` 读取 Jira snapshot 并标准化 bug 主体信息。
2. `project-context` 结合配置与只读输入完成项目、需求、仓库、模块上下文解析。
3. `requirement-summarizer` 生成 `RequirementBrief`。
4. `code-locator` 生成代码候选、影响模块和根因假设。
5. `fix-planner` 生成修复建议、影响范围和验证建议。
6. `verification-recorder` 标准化外部补录的验证信息。
7. `gitlab-linker` 与 `feishu-recorder` 输出 canonical draft。
8. `report-writer` 汇总最终报告输入。

### 6.2 输入与输出

输入：

- Jira snapshot
- `ProjectProfile`
- repo candidate modules
- GitLab artifact 标准结构
- 外部 verification 信息

输出：

- `StageResult<RequirementBrief>`
- `StageResult<CodeLocationResult>`
- `StageResult<FixPlan>`
- `StageResult<VerificationRecord>`
- `StageResult<CanonicalDraft>`
- `StageResult<BugfixReport>`

### 6.3 关键约束

- 每个 skill 都必须无状态。
- 每个 skill 都必须有清晰错误语义，便于 workflow 判断等待输入、人工修正或终止失败。
- 分析产物不得依赖 CLI 渲染才能继续推进。
- canonical draft 必须兼容后续 preview、`preview_hash`、marker 和幂等键生成。

### 6.4 异常与失败分支

- 输入缺失：返回 `invalid_input`。
- 配置缺失或规则冲突：返回结构化配置错误。
- 需求多候选或无法识别：返回结构化分支结果，不得静默忽略。
- 外部读取结果不完整：返回可追踪错误并携带 `source_refs`。

## 7. 开发任务拆解

1. 定义 `StageResult<T>` 与通用错误壳。
2. 定义各 skill 的输入输出结构。
3. 定义 `RequirementBrief`、代码定位结果、修复计划和验证建议的最小字段。
4. 定义需求映射与人工覆盖的结构化表达。
5. 定义 Jira / Feishu canonical draft 的输入结构。
6. 定义 report-writer 的汇总输入与输出边界。

## 8. 测试与验收

单元/契约测试：

- `StageResult<T>` 字段完整性
- `RequirementBrief` 最小字段集
- 需求映射四类分支
- 修复计划与验证建议输出
- canonical draft 一致性

集成测试：

- 只读输入可以被连续转化为 `RequirementBrief`、代码定位和修复计划
- report-writer 可在不依赖渲染层的情况下输出结构化报告内容

文档一致性检查：

- 不把审批、渲染和 connector 路由塞入 skill
- 不引入未授权的自动改代码能力

验收标准：

- F05 与 F06 可以直接消费 F04 产物
- 所有分析链路结果都具备稳定字段和错误语义

## 9. 与其他 Feature 的交接点

- 从 F03 获取只读输入。
- 向 F05 提供审批前的分析产物。
- 向 F06 提供 preview 生成的 canonical draft。
- 向 F08 提供 `BugfixReport` 的主要内容来源。
