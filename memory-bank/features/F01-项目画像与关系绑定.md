# F01 项目画像与关系绑定

## 1. 目标与用户价值

本 feature 负责建立项目画像配置、平台绑定关系和需求映射规则，使系统具备一个唯一可信的运行前提。它直接服务于“显式配置优先”和“缺失信息必须补录”的核心原则。

用户价值如下：

- 用户可以通过显式配置维护项目与 Jira、GitLab、飞书、本地仓库之间的关系。
- agent 在运行时可以稳定加载同一份 `ProjectProfile`，避免高风险静默推断。
- 需求映射失败、多候选需求和“未绑定需求”场景都能获得统一处理策略。

## 2. 来源依据

### 2.1 已确认事实

- 系统必须维护一份项目画像配置，作为唯一可信来源。
- 项目画像至少覆盖 Jira、GitLab、飞书、repo、审批、序列化与脱敏策略。
- 需求映射规则必须支持多来源、优先级、人工指定、多候选确认和“未绑定需求”降级。

### 2.2 上游约束

- 需求文档第 9 章定义了项目关系绑定的最小字段要求和未绑定需求策略。
- 技术方案要求 `ProjectProfile` 位于核心领域模型中，并通过 `config-loader` 进入运行时。
- 实施计划将该能力纳入任务一与任务七的共同前提，不允许在 workflow 内临时猜测配置。

### 2.3 本 Feature 实现决策

- `ProjectProfile` 作为唯一项目配置对象，不再引入平行配置模型。
- 需求映射规则和未绑定需求策略与配置一起保存，不下沉到 workflow 的自由文本判断。
- CLI 只负责配置维护入口，不拥有配置语义真源。

## 3. 范围

### 3.1 本 Feature 要做什么

- 定义 `ProjectProfile` 的字段结构、版本和最小校验要求。
- 设计 Jira / GitLab / 飞书 / repo 的关系绑定信息。
- 定义需求来源、映射规则、回退动作和人工覆盖方式。
- 定义“未绑定需求”的显式表达方式。
- 为后续 `bind` 和 `inspect config` 提供稳定输入输出基础。

### 3.2 本 Feature 明确不做什么

- 不管理 run 生命周期、checkpoint、side-effect ledger。
- 不负责 preview、execute、幂等去重或 reconcile。
- 不直接连接外部系统读取真实数据。
- 不定义审批状态机和恢复逻辑。

## 4. 前置依赖与输出产物

前置依赖：

- 无代码前置依赖，但必须遵守需求文档、技术方案和实施计划中的 v1 范围约束。

输出产物：

- `ProjectProfile` 结构定义
- 需求映射规则说明
- 未绑定需求处理规则
- 绑定命令所需输入语义
- 后续 feature 可复用的配置读取契约

## 5. 涉及的核心对象、状态与能力边界

核心对象：

- `ProjectProfile`
- `StructuredError`

相关状态或字段：

- `config_version`
- `approval_policy`
- `serialization_policy`
- `sensitivity_policy`
- `requirements.source_type`
- `requirement_binding_status`
- `binding_reason`

能力边界：

- 本 feature 只负责“配置与规则真源”。
- workflow 只消费配置，不解释配置来源。
- connector 只消费平台配置，不修改绑定策略。

## 6. 实现设计

### 6.1 主要流程

主要流程如下：

1. 用户通过 CLI 维护项目画像配置。
2. 系统对最小字段进行结构化校验。
3. 运行时通过 `config-loader` 读取同一份配置。
4. `project-context` 依据映射规则解析目标需求。
5. 若无法唯一命中，则进入人工指定、多候选确认或“未绑定需求”分支。

### 6.2 输入与输出

输入：

- 项目标识与项目名称
- Jira / GitLab / 飞书 / repo 配置
- requirement link rules
- 审批、序列化、脱敏策略

输出：

- 可持久化的 `ProjectProfile`
- 需求映射规则集合
- 明确的未绑定需求策略
- 供 F03、F04、F07 复用的配置读取输入

### 6.3 关键约束

- 配置缺失时必须提示补录，不能静默猜测。
- 不同项目之间的凭证引用、模板和规则必须隔离。
- 未绑定需求允许继续分析，但是否允许进入写回由项目配置控制。
- `ProjectProfile` 只保存 `credential_ref`，不保存明文凭证。

### 6.4 异常与失败分支

- 缺失最小字段：返回 `invalid_input`。
- 配置版本不匹配：返回结构化配置错误，阻止继续运行。
- 需求映射失败：根据规则进入阻塞、人工指定或降级为“未绑定需求”。
- 多候选需求：不得自动选定，必须进入人工确认或人工覆盖。

## 7. 开发任务拆解

1. 定义 `ProjectProfile` 的 schema、字段分组和最小必填集合。
2. 定义需求映射规则的数据结构和优先级表达。
3. 定义“未绑定需求”的结构化表达与传播字段。
4. 定义配置读取与配置校验的统一错误出口。
5. 定义 CLI 绑定命令需要接受和输出的配置语义。
6. 为配置查看、配置完整度检查和需求映射规则检查准备可复用输出格式。

## 8. 测试与验收

单元/契约测试：

- `ProjectProfile` 最小字段校验
- requirement link rules 结构校验
- 多候选需求与人工覆盖分支校验
- 未绑定需求字段传播校验

集成测试：

- `bind` 命令维护配置后可被运行时加载
- `inspect config` 能显示关键绑定信息与缺失项

文档一致性检查：

- 不引入 v1 非目标
- 不把需求映射策略散落到 workflow
- 与 F03、F04、F07 的接口边界一致

验收标准：

- 存在唯一可信配置入口
- 缺失配置能被显式发现
- 未绑定需求策略可被后续 feature 一致复用

## 9. 与其他 Feature 的交接点

- 向 F02 提供持久化配置入口，但不承担 run 存储。
- 向 F03 提供 connector 选择所需的平台配置。
- 向 F04 提供需求映射规则与未绑定需求表达。
- 向 F07 提供 `bind` / `inspect` 命令的配置语义。
