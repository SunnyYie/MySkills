# Bugfix Orchestrator v1 实施计划

> 面向 AI 开发人员：按顺序执行本计划。每次只完成一个步骤；每完成一个步骤都必须立即完成该步骤对应测试；测试未通过时不得进入下一步；本文件只提供实施说明，不提供代码片段。

## 1. 文档目标

本文档基于 [需求文档](/Users/sunyi/ai/MySkills/memory-bank/需求文档.md) 与 [技术方案](/Users/sunyi/ai/MySkills/memory-bank/技术方案.md)，将 `Bugfix Orchestrator v1` 拆解为可直接执行的实施步骤。

本文档的目标不是重复需求或技术方案，而是给 AI 开发人员一组小而具体、可验证、可交付的实施指令。

## 2. 交付目标

v1 需要交付一条本地 `CLI-first` bugfix 编排闭环，覆盖以下能力：

- Jira bug intake
- 项目、仓库、需求与模块上下文解析
- `Requirement Brief` 生成与审批
- 代码定位建议与修复计划生成
- 外部修复产物与验证结果补录
- Jira 回写预览与执行
- 飞书记录预览与执行
- `Bugfix Report` 导出
- checkpoint 持久化、恢复执行、审计追踪与脱敏落盘

## 3. 范围边界

### 3.1 v1 必须包含

- 项目画像配置维护
- CLI 命令体系
- 八阶段主工作流
- 三个子工作流
- Skill 体系与结构化产物
- Jira、GitLab、飞书、本地仓库四类基础设施适配
- 审批、回退、恢复与副作用去重
- 报告导出与验收闭环

### 3.2 v1 明确不包含

- 自动修改代码
- 自动执行测试
- 自动创建 commit、branch、MR
- 服务端多用户调度
- 自然语言对话式入口
- 复杂凭证托管平台
- 高自治知识图谱推理

## 4. 实施总规则

- `Workflow/Agent Layer` 是唯一业务状态 owner。
- `CLI Layer` 只负责命令入口、交互、展示和参数透传。
- `Skill Layer` 只接受结构化输入并返回结构化输出，不做 I/O，不持久化 run state。
- `Infrastructure Layer` 是唯一外部系统与本地仓库访问入口。
- 所有外部写操作都必须遵守 `preview -> approve -> execute`。
- 所有关键状态迁移都必须落 checkpoint。
- 恢复执行只能依赖持久化状态，不能依赖内存态。
- 默认以显式配置优先，避免高风险静默推断。
- 敏感信息不得明文落盘。

## 5. 执行方式

### 5.1 完成定义

每个步骤只有在以下条件同时满足时才算完成：

- 该步骤要求的实现或文档变更已经完成。
- 该步骤要求的测试已经执行。
- 测试结果满足本步骤给出的通过标准。
- 未引入与需求文档、技术方案冲突的新约束。

### 5.2 测试规则

- 每个步骤都必须有测试，不允许把测试集中到阶段末尾统一补做。
- 优先增加最小范围测试，避免一次测试覆盖多个尚未稳定的能力。
- 测试说明必须包含测试对象、触发方式、预期结果和失败判定。
- 如果某步骤是文档或契约冻结步骤，也必须做一致性测试，验证其与需求文档、技术方案、已有计划内容不存在冲突。

## 6. 分步实施说明

### 任务一：锁定领域边界与核心契约

**目标：** 在开始编码前，先冻结 v1 的边界、分层职责、核心对象和状态语义，避免后续返工。

### 步骤 1.1：核对 v1 范围与非范围

- 执行动作：逐条比对需求文档与技术方案中的“覆盖范围”和“非目标”，在实现说明中确认 v1 必交付能力与明确排除项。
- 测试：做一次范围一致性检查，确认实施计划中的“必须包含”和“明确不包含”与需求文档、技术方案没有冲突，也没有新增未评审能力。
- 通过标准：v1 范围表述完整，且不存在“需求文档允许、实施计划遗漏”或“实施计划新增、需求文档未授权”的内容。

### 步骤 1.2：冻结系统分层职责

- 执行动作：明确 `CLI Layer`、`Workflow/Agent Layer`、`Skill Layer`、`Infrastructure Layer`、`Renderers` 的责任边界和禁止行为。
- 测试：做一次职责冲突检查，逐层验证是否存在状态 ownership 重叠、越层 I/O 或重复渲染责任。
- 通过标准：每一层都有唯一职责描述，且关键职责不存在双 owner。

### 步骤 1.3：冻结核心对象清单

- 执行动作：确认 `ProjectProfile`、`ExecutionContext`、`RequirementBrief`、`BugfixReport`、`ApprovalRecord`、`SideEffectLedgerEntry`、`CheckpointRecord`、`StructuredError` 为 v1 核心对象。
- 测试：做一次对象覆盖性检查，验证需求中的核心输入、核心产物、审批记录、恢复记录和错误语义都能映射到上述对象。
- 通过标准：不存在无法归属的核心信息，也不存在同一业务信息被两个对象重复持有。

### 步骤 1.4：冻结状态枚举与兼容映射

- 执行动作：确定阶段状态、审批状态、审批决定、`run_lifecycle_status`、`run_outcome_status` 以及兼容字段 `run_status` 的映射规则。
- 测试：做一次状态完备性检查，覆盖标准主流程、审批拒绝、回退重做、部分成功、恢复执行五类场景。
- 通过标准：所有关键场景都能得到唯一、无歧义的状态组合，且 `run_status` 派生规则稳定。

### 步骤 1.5：冻结审批对象分型规则

- 执行动作：明确分析产物审批与外部写入审批必须使用不同绑定对象，前者绑定 `artifact_ref`，后者绑定 `preview_ref + preview_hash`。
- 测试：做一次审批对象混用检查，验证分析阶段不会错误依赖 preview，写入阶段不会绕过 preview 直接审批。
- 通过标准：两类审批的绑定对象、失效条件和后续动作完全分离。

### 步骤 1.6：冻结条件必填与错误语义

- 执行动作：明确 GitLab 产物、Jira 写回、飞书写入、未绑定需求、权限错误、状态错误、对账失败等场景的最小字段要求与标准错误类别。
- 测试：做一次错误语义覆盖检查，验证每类输入缺失或非法状态都有唯一的错误归属与响应口径。
- 通过标准：不存在“相同错误多种命名”或“错误发生但无标准响应”的情况。

### 任务二：建立持久化、审计与安全底座

**目标：** 先让系统具备可恢复、可追踪、可脱敏能力，再承载上层业务流程。

### 步骤 2.1：冻结项目配置与 run 存储布局

- 执行动作：确定项目画像存储位置、run 根目录、`context.json`、`events.ndjson`、`side-effects.ndjson`、`checkpoints/`、`artifacts/`、`lock` 的目录结构与用途。
- 测试：做一次存储布局检查，验证每类运行期信息都有唯一落位，且不存在“同一类信息散落多个位置”的问题。
- 通过标准：目录结构完整、职责清晰，并能支撑恢复、审计与产物引用。

### 步骤 2.2：冻结落盘 allowlist 与脱敏规则

- 执行动作：明确 `ExecutionContext`、checkpoint、artifact、报告和日志中哪些字段允许落盘，哪些字段必须脱敏或只保留引用。
- 测试：做一次敏感信息落盘检查，覆盖 token、cookie、Authorization、原始 payload、自由文本正文、审计细节等高风险内容。
- 通过标准：敏感字段都被拦截或脱敏，恢复执行所需最小信息仍然保留。

### 步骤 2.3：冻结 checkpoint 触发点

- 执行动作：列出所有必须落 checkpoint 的关键状态迁移，包括阶段开始、阶段完成、等待审批、等待外部输入、进入副作用执行前、恢复执行后等节点。
- 测试：做一次恢复路径检查，验证任一中断点之后都存在足够的持久化状态用于恢复，不依赖内存态。
- 通过标准：不存在“状态已变化但没有可恢复检查点”的空窗区间。

### 步骤 2.4：冻结 side-effect ledger 生命周期

- 执行动作：明确外部写入必须遵守 `prepared -> executing_side_effect checkpoint -> dispatched -> terminal status` 顺序，并定义 `succeeded`、`failed`、`outcome_unknown` 的落账条件。
- 测试：做一次副作用顺序检查，验证写请求发出前一定已经记录 `prepared` 并完成对应 checkpoint。
- 通过标准：所有外部写入都具备可追踪账本链路，且顺序不可倒置。

### 步骤 2.5：冻结锁、去重与对账基线

- 执行动作：确定 run 级锁、target 级锁或 dedupe index、reconcile 触发条件及“未知结果”收敛规则。
- 测试：做一次并发与恢复冲突检查，覆盖重复写入、跨 run 冲突、`prepared` 或 `dispatched` 无终态记录三类场景。
- 通过标准：重复副作用可被阻止或识别，未知结果场景必须先对账再决定是否重试。

### 任务三：接入 Infrastructure 只读能力

**目标：** 先打通外部读取与本地只读能力，为 workflow 和 skill 提供稳定输入。

### 步骤 3.1：冻结能力接口清单

- 执行动作：确认 `HealthCheckCapability`、`ReaderCapability`、`TargetResolverCapability`、`PreviewWriterCapability`、`SideEffectExecutorCapability`、`ArtifactResolverCapability`、`RepoWorkspaceCapability` 的职责与边界。
- 测试：做一次接口职责检查，验证读取、目标解析、preview、执行、副作用对账和仓库只读访问没有责任重叠。
- 通过标准：每个能力都有唯一用途，且能覆盖技术方案中的基础设施责任。

### 步骤 3.2：冻结 Jira 只读契约

- 执行动作：明确 Jira connector 需要提供的 bug 读取字段、需求线索读取方式、写回目标解析结果以及错误标准化输出。
- 测试：做一次 Jira 输入覆盖检查，确认 issue 状态、描述、标签、需求线索、写回目标都可被结构化读取。
- 通过标准：Jira 只读数据足以支撑 `jira-intake`、`project-context` 和后续写回预览。

### 步骤 3.3：冻结 GitLab 只读契约

- 执行动作：明确 GitLab connector 需要标准化的 commit、branch、MR 引用格式，以及项目信息、默认分支、链接生成规则。
- 测试：做一次 GitLab 产物标准化检查，验证三类产物都能被解析为统一结构，且后续可用于 Jira 回写与报告导出。
- 通过标准：GitLab 产物表示稳定，不依赖调用方自己拼接链接。

### 步骤 3.4：冻结 Feishu 与本地仓库只读契约

- 执行动作：明确 Feishu 目标定位、模板读取、锚点解析、本地仓库信息、模块候选解析和只读搜索能力。
- 测试：做一次目标定位检查，验证飞书目标位置与仓库模块候选都能被稳定解析，不需要 workflow 越层处理底层细节。
- 通过标准：Feishu 与 repo 的只读输入已经足以支持 preview 生成和代码定位分析。

### 步骤 3.5：冻结 connector registry 与路由规则

- 执行动作：定义如何根据 `ProjectProfile` 和所需 capability 选择 connector，并禁止 workflow 直接依赖具体平台实现。
- 测试：做一次路由一致性检查，验证同一项目在不同阶段获得的 connector 选择结果一致，且不会绕过 registry。
- 通过标准：平台选择规则单一明确，可复用且可测试。

### 任务四：落地 Skill 层契约

**目标：** 将分析能力与文档能力收敛为无状态、可复用、可独立测试的 skill 集合。

### 步骤 4.1：冻结 skill 清单与职责

- 执行动作：确认 `config-loader`、`jira-intake`、`project-context`、`requirement-summarizer`、`code-locator`、`fix-planner`、`verification-recorder`、`gitlab-linker`、`feishu-recorder`、`report-writer` 的唯一职责。
- 测试：做一次 skill 边界检查，验证审批、渲染、connector 路由、持久化等责任没有被错误塞入 skill。
- 通过标准：每个 skill 都是“结构化输入 -> 结构化输出”的纯能力单元。

### 步骤 4.2：冻结统一输出封装

- 执行动作：统一所有 skill 输出 `StageResult<T>`，至少包含 `status`、`summary`、`data`、`warnings`、`errors`、`source_refs`、`generated_at`。
- 测试：做一次输出一致性检查，验证所有 skill 都能以相同外壳被 workflow 消费，不需要额外分支兼容。
- 通过标准：不存在某个 skill 需要特殊解析逻辑才能接入 workflow。

### 步骤 4.3：冻结需求映射与人工覆盖规则

- 执行动作：明确 `project-context` 对 `unresolved`、多候选需求、人工指定和人工覆盖的处理结果与状态表达。
- 测试：做一次需求映射分支检查，覆盖唯一命中、多候选、无法识别、人工覆盖四种场景。
- 通过标准：每种场景都能生成稳定的结构化结果，并给出后续允许动作。

### 步骤 4.4：冻结分析类产物最小字段集

- 执行动作：明确 `RequirementBrief`、代码定位结果、修复计划、验证建议、外部补录验证结果的最小字段与摘要要求。
- 测试：做一次产物完整性检查，验证这些结构化结果足以驱动审批、回退、报告和写回草稿生成。
- 通过标准：不存在必须依赖原始自由文本才能继续推进的分析步骤。

### 步骤 4.5：冻结写入类 canonical draft 规则

- 执行动作：明确 `gitlab-linker` 与 `feishu-recorder` 输出的 canonical draft 必须兼容 preview 生成、preview hash、marker 和幂等键生成。
- 测试：做一次写入草稿一致性检查，验证同一业务输入在 dry-run 与真实写入链路中能得到一致的预览基础数据。
- 通过标准：写入草稿可作为 preview 和 execute 的共同输入，不需要二次拼装。

### 步骤 4.6：冻结 skill 错误语义

- 执行动作：为每个 skill 明确输入缺失、依赖数据无效、候选歧义、配置缺失、外部读取失败等错误输出方式。
- 测试：做一次错误出口检查，验证 workflow 能区分“继续等待输入”“需要人工修正”“直接失败终止”三类结果。
- 通过标准：skill 错误语义可直接驱动状态机，而不是依赖人工猜测。

### 任务五：实现主工作流与状态机

**目标：** 完成 v1 的八阶段主流程、审批门、回退、恢复和外部输入等待机制。

### 步骤 5.1：冻结八阶段主流程

- 执行动作：按顺序定义 `Intake`、`Context Resolution`、`Requirement Synthesis`、`Code Localization`、`Fix Planning`、`Execution`、`Artifact Linking`、`Knowledge Recording` 的进入条件、输入和输出。
- 测试：做一次主流程完整性检查，验证从 Jira issue 到报告导出之间不存在缺口阶段或重复阶段。
- 通过标准：每个阶段的输入来源、输出产物和下一步去向明确。

### 步骤 5.2：冻结审批门与审批结果语义

- 执行动作：明确哪些阶段必须审批，`approve`、`reject`、`revise` 在分析阶段和写入阶段分别触发什么状态变化。
- 测试：做一次审批流转检查，覆盖“通过继续”“拒绝终止”“退回重做”“写入已批但尚未执行”四类结果。
- 通过标准：审批结果语义稳定，且“审批通过”与“真实写入执行”严格分离。

### 步骤 5.3：冻结 `Execution` 阶段输入完成矩阵

- 执行动作：明确 GitLab 产物补录、验证结果补录、最小前置输入、幂等合并与替换规则。
- 测试：做一次外部输入矩阵检查，覆盖仅补录 artifact、仅补录 verification、两者齐备、重复补录四种情况。
- 通过标准：`Execution` 阶段能稳定判断是继续等待、允许推进还是更新当前有效摘要。

### 步骤 5.4：冻结 `stale` 与 `superseded` 规则

- 执行动作：明确 revise 后旧 artifact、旧审批、旧 preview 的失效条件，以及哪些历史记录必须保留但不能再参与恢复。
- 测试：做一次回退后有效态检查，验证旧结果不会再次被当作当前输入，但历史审计仍可追溯。
- 通过标准：回退后的“当前有效态”和“历史事实”分离清晰。

### 步骤 5.5：冻结恢复执行规则

- 执行动作：明确最近 checkpoint 恢复、显式 `--checkpoint` 恢复、`waiting_external_input` 恢复、未终态副作用恢复和 `outcome_unknown` 恢复规则。
- 测试：做一次恢复分支检查，覆盖中断后重进、指定 checkpoint 回滚、恢复前先 reconcile 三类路径。
- 通过标准：恢复动作不会静默重跑已批准分析阶段，也不会重复执行已成功副作用。

### 步骤 5.6：冻结 `partial_success` 语义

- 执行动作：明确至少一个外部写成功且至少一个不可忽略副作用失败时，run 应如何记录最终结果、允许哪些后续操作。
- 测试：做一次部分成功检查，验证系统能保留成功副作用、标记失败副作用，并允许只重试失败部分。
- 通过标准：`partial_success` 既不是完全成功，也不会退化为不可恢复的整体失败。

### 任务六：闭环写入 preview、审批与真实执行

**目标：** 让 Jira 与飞书写入具备“可预览、可审批、可去重、可恢复”的完整执行链路。

### 步骤 6.1：冻结 preview 生成规则

- 执行动作：明确 Jira 回写与飞书写入如何从 canonical draft 生成 preview，以及 preview 中哪些字段允许变化、哪些字段必须稳定。
- 测试：做一次 preview 完整性检查，验证 preview 足以让用户判断是否应执行真实写入。
- 通过标准：preview 信息可读、可审批，且与真实写入 payload 只有动态字段差异。

### 步骤 6.2：冻结 preview 版本与审批绑定

- 执行动作：明确 preview 刷新后的新 `preview_ref`、新 `preview_hash`、旧审批失效规则以及执行前一致性校验要求。
- 测试：做一次旧 preview 失效检查，验证 preview 被刷新后，旧审批不能继续用于真实写入。
- 通过标准：审批永远绑定当前有效 preview 版本，无法绕过版本校验。

### 步骤 6.3：冻结 execute 前置条件

- 执行动作：明确真实写入前必须满足的条件，包括审批通过、preview 一致、目标查重通过、账本准备完成、checkpoint 已落盘。
- 测试：做一次执行前门禁检查，分别验证缺审批、preview 不一致、重复目标、未落账四种情况下都会被阻止。
- 通过标准：真实写入只会在前置条件全部满足时触发。

### 步骤 6.4：冻结 marker、幂等键与去重规则

- 执行动作：为 Jira 与飞书写入定义稳定 marker、target ref 和幂等键生成口径，统一 dry-run、resume、record 和主流程使用方式。
- 测试：做一次重复写入检查，验证相同业务写入请求在重试、恢复、跨命令入口下都不会产生重复副作用。
- 通过标准：去重逻辑统一，不依赖调用入口差异。

### 步骤 6.5：冻结 non-interactive 与 dry-run 语义

- 执行动作：明确 `--non-interactive` 下必须显式确认 `preview_hash`，以及 dry-run 允许读取与 preview、禁止真实写入和成功副作用记录。
- 测试：做一次模式差异检查，覆盖交互式、非交互式、dry-run 三种执行模式。
- 通过标准：不同运行模式的允许动作、失败条件和输出差异清晰且可预测。

### 步骤 6.6：冻结对账与未知结果收敛规则

- 执行动作：明确 `confirmed_applied`、`confirmed_not_applied`、`still_unknown` 三类 reconcile 结果以及对应后续动作。
- 测试：做一次未知结果恢复检查，验证 `outcome_unknown` 场景下必须先对账，再决定是否可重试。
- 通过标准：未知结果不会被直接当作失败重放，也不会被错误视为成功。

### 任务七：补齐 CLI 与子工作流

**目标：** 通过统一 CLI 暴露所有能力，同时保证子工作流复用同一状态机，不形成旁路。

### 步骤 7.1：冻结 CLI 命令树

- 执行动作：明确 `bind`、`inspect`、`run`、`record` 四组命令的职责边界和典型入口。
- 测试：做一次命令分组检查，验证每个命令只负责一种类型的行为，不与其他命令组重复。
- 通过标准：CLI 命令树清晰，用户可以从命令名称直接判断行为范围。

### 步骤 7.2：冻结 `bind` 与 `inspect` 语义

- 执行动作：明确项目绑定、连接器检查、运行态检查、配置查看和图谱查看的输入参数与输出内容。
- 测试：做一次配置可观测性检查，验证用户可以在正式运行前确认配置完整度与连接器可用性。
- 通过标准：配置问题能在进入主流程前被发现，而不是等到执行中才暴露。

### 步骤 7.3：冻结 `run` 命令行为

- 执行动作：明确 `run start`、`run brief`、`run resume`、`run status`、`run approve`、`run revise`、`run reject`、`run provide-artifact`、`run provide-verification`、`run preview-write`、`run execute-write` 的输入、触发条件和输出。
- 测试：做一次主命令可达性检查，验证标准主流程中的每个关键动作都有唯一命令入口。
- 通过标准：主流程不需要借助隐式操作或手工改文件才能推进。

### 步骤 7.4：冻结 `record` 子工作流

- 执行动作：明确“仅生成 Requirement Brief”“仅回写 Jira”“仅写入飞书记录”三类子工作流的最小输入、补录方式和审批要求。
- 测试：做一次子工作流旁路检查，验证 `record` 仍然复用统一状态机、统一 preview 机制和统一审批规则。
- 通过标准：`record` 是快捷入口，不是绕过工作流约束的后门。

### 步骤 7.5：冻结 CLI 输出契约

- 执行动作：明确 TTY 输出、JSON 输出、错误输出与 `status --json` 最小字段集合。
- 测试：做一次输出契约检查，验证状态字段、允许动作、等待原因、所需输入、当前 preview 引用等信息都可被稳定输出。
- 通过标准：CLI 输出与状态机语义一致，调用方不需要猜测内部状态。

### 步骤 7.6：冻结标准演示路径

- 执行动作：将 `bind -> inspect -> run -> approve/revise -> dry-run writeback -> real writeback -> export report` 固化为演示路径。
- 测试：做一次端到端路径检查，验证该路径能覆盖配置、分析、审批、补录、预览、执行、报告导出全链路。
- 通过标准：存在一条单一、可重复执行、可验证的 v1 标准演示路径。

### 任务八：完成报告、验收与交付收口

**目标：** 让每条需求都能映射到可执行验证路径，并形成最终交付包。

### 步骤 8.1：冻结 `BugfixReport` 最小内容

- 执行动作：确认报告必须包含 Bug 基本信息、关联需求、代码定位结果、修复方案摘要、验证结果、GitLab 链接、Jira 回写摘要、飞书记录摘要、审批历史、开放风险和最终状态。
- 测试：做一次报告覆盖检查，验证报告可以作为 run 的统一最终输出，不需要回头拼接多份中间产物。
- 通过标准：报告字段足以支撑复盘、审计和交付验收。

### 步骤 8.2：建立需求到测试的追踪矩阵

- 执行动作：为每条验收场景建立“需求/验收场景 ID、对应阶段、测试层级、前置数据、执行动作、预期退出码、预期状态、证据产物路径、失败判定”的追踪关系。
- 测试：做一次需求覆盖率检查，验证需求文档中的所有 v1 验收场景都至少对应一条验证路径。
- 通过标准：不存在“需求存在，但没有验证方式”或“有测试，但无法说明验证的是哪条需求”的情况。

### 步骤 8.3：冻结阶段性测试套件结构

- 执行动作：将测试组织为领域模型、持久化与安全、skill、workflow、connector 契约、CLI 集成、需求追踪与指标验证、验收场景八类。
- 测试：做一次测试分层检查，验证单元测试、契约测试、集成测试、验收测试分工清楚，没有重复覆盖或明显缺口。
- 通过标准：测试结构能支撑“先约束、后流程、再集成”的实施顺序。

### 步骤 8.4：冻结成功指标与证据要求

- 执行动作：明确首次绑定时长、`Requirement Brief` 生成耗时、主流程成功率、恢复执行成功率、dry-run 覆盖率的统计口径、样本范围与阈值门禁。
- 测试：做一次指标可统计性检查，验证每个指标都能从运行产物和审计记录中计算出来。
- 通过标准：成功指标不是口号，而是可采样、可计算、可复核的数字。

### 步骤 8.5：完成最终验收清单

- 执行动作：将标准主流程、审批分支、配置缺失补录、dry-run 与真实写入、中断恢复、部分成功与重试去重、需求映射歧义、子工作流独立完成纳入最终验收清单。
- 测试：做一次验收场景完备性检查，验证需求文档列出的典型场景全部可被最终清单覆盖。
- 通过标准：v1 所有关键场景都有明确验收入口、验证动作和证据要求。

### 步骤 8.6：完成交付包清单

- 执行动作：确认最终交付必须包含可维护的项目画像配置结构、可运行 CLI、八阶段工作流、三类子工作流、结构化 skill、四类适配能力、checkpoint 与账本机制、`Requirement Brief`、写回 preview 与执行能力、`BugfixReport`、使用与验收文档。
- 测试：做一次交付物对照检查，验证交付包与实施目标逐项对应，没有“已实现但未交付说明”或“写入交付物但实际无实现”的项。
- 通过标准：交付包完整、可验收、可演示。

## 7. 推荐实施顺序

严格按以下顺序推进：

1. 任务一：锁定领域边界与核心契约
2. 任务二：建立持久化、审计与安全底座
3. 任务三：接入 Infrastructure 只读能力
4. 任务四：落地 Skill 层契约
5. 任务五：实现主工作流与状态机
6. 任务六：闭环写入 preview、审批与真实执行
7. 任务七：补齐 CLI 与子工作流
8. 任务八：完成报告、验收与交付收口

该顺序不可随意调整，原因如下：

- 先锁定契约和状态机，避免后续返工。
- 先做持久化与恢复，避免流程能力建好后再倒补基础设施。
- 先打通只读分析链路，再接外部写入，降低副作用风险。
- 将最复杂的恢复与去重逻辑建立在统一账本之上。

## 8. 重点风险

- 状态机、审批状态与 run 状态混用。
- 分析审批与写入审批对象混用。
- preview 刷新后旧审批失效处理不一致。
- 恢复执行时重复触发外部写入。
- `outcome_unknown` 场景下误重试。
- reconcile 能力不足导致结果未知无法收敛。
- 未绑定需求场景推进策略不一致。
- `ExecutionContext` 落盘过重导致敏感信息泄露或恢复复杂化。
- `record` 命令演变为旁路流程。
- 成功指标缺少量化验证导致“可运行但不可验收”。

## 9. 默认假设

- 当前目标是先交付稳定、可解释、可恢复的闭环，而不是全自动 agent。
- 项目关系绑定以显式配置为主。
- 关键节点必须支持人工确认。
- 未绑定需求默认允许继续分析，但在强约束项目中进入外部写回前必须阻塞。
- 所有外部写操作都必须具备 preview、审批、checkpoint 与幂等去重能力。
- `Workflow/Agent Layer` 是唯一业务状态 owner。
- 本文档仅提供实施说明，不包含代码实现内容。
