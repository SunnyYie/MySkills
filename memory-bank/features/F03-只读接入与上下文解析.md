# F03 只读接入与上下文解析

## 1. 目标与用户价值

本 feature 负责打通 Jira、GitLab、飞书和本地仓库的只读接入，并为后续分析、写回预览和代码定位提供统一输入。

用户价值如下：

- 系统能从 Jira issue 出发获得稳定的 bug 上下文。
- GitLab 产物和项目元数据具备统一表达。
- 飞书目标位置与模板可被解析为预览输入。
- 本地仓库与模块候选可被结构化读取，而不是靠 workflow 直接搜索。

## 2. 来源依据

### 2.1 已确认事实

- Infrastructure Layer 是唯一外部系统与本地仓库访问入口。
- connector registry 必须根据 `ProjectProfile + capability` 进行路由。
- repo workspace 负责 Git 元数据和代码候选定位的只读支持。

### 2.2 上游约束

- 技术方案将 connector、repo workspace、registry 划归 `src/infrastructure`。
- 实施计划任务三冻结了能力接口、Jira/GitLab/Feishu/repo 的只读契约和路由规则。
- workflow 与 skill 不得越层直接访问外部系统或仓库。

### 2.3 本 Feature 实现决策

- 只读接入与写入执行分离；本 feature 不承担 preview 或 execute。
- 所有只读结果都转成标准化结构，避免调用方自己拼装字段或链接。
- connector 选择统一通过 registry，不允许平行平台选择逻辑。

## 3. 范围

### 3.1 本 Feature 要做什么

- 定义 `HealthCheckCapability`、`ReaderCapability`、`TargetResolverCapability`、`ArtifactResolverCapability`、`RepoWorkspaceCapability` 等只读接口边界。
- 定义 Jira bug 读取字段、需求线索、写回目标解析输入。
- 定义 GitLab commit / branch / MR 统一 artifact 表示。
- 定义 Feishu 目标定位、模板读取、锚点解析输入。
- 定义 repo workspace 的模块候选、Git 元数据和只读搜索输出。

### 3.2 本 Feature 明确不做什么

- 不生成 preview。
- 不执行真实外部写入。
- 不决定审批或状态流转。
- 不生成分析类业务产物。

## 4. 前置依赖与输出产物

前置依赖：

- F01 提供平台配置与绑定规则。

输出产物：

- connector 能力接口定义
- 标准化 Jira / GitLab / Feishu / repo 只读输出
- registry 路由规则
- 为 F04、F06 提供统一输入结构

## 5. 涉及的核心对象、状态与能力边界

核心对象：

- `ProjectProfile`
- `StructuredError`
- GitLab artifact 标准结构

能力边界：

- `ReaderCapability` 负责外部读取，不负责业务分析。
- `RepoWorkspaceCapability` 提供仓库只读搜索，不直接判断根因。
- `TargetResolverCapability` 可以解析目标位置，但不负责最终写入。

## 6. 实现设计

### 6.1 主要流程

主要流程如下：

1. workflow 或 skill 根据 `ProjectProfile + capability` 从 registry 获取 connector。
2. Jira connector 拉取 issue 主体、状态、标签、描述和需求线索。
3. GitLab connector 标准化外部产物和项目信息。
4. Feishu connector 解析文档目标与模板位置。
5. repo workspace 读取本地仓库、模块规则和搜索结果摘要。

### 6.2 输入与输出

输入：

- `ProjectProfile`
- Jira issue key
- repo local path 与 module rules
- 飞书文档与模板定位信息

输出：

- Jira issue snapshot
- 标准化 GitLab artifact 与项目元数据
- Feishu target snapshot
- repo candidate modules 与搜索命中摘要

### 6.3 关键约束

- 链接生成属于 connector 职责，不得散落到 workflow、skill 或 renderer。
- workflow 不能直接访问 Jira、GitLab、飞书或 repo。
- registry 的解析输入只能是 `ProjectProfile + capability`。
- 只读输出必须足以支撑 `jira-intake`、`project-context`、preview 目标解析和代码定位分析。

### 6.4 异常与失败分支

- connector 不可用：返回 capability 不满足错误。
- issue 不存在或权限不足：返回结构化读取错误。
- repo 路径无效或模块规则非法：返回结构化输入错误。
- 飞书目标无法定位：返回目标解析错误，阻止后续 preview 生成。

## 7. 开发任务拆解

1. 定义 connector 能力接口与 registry 解析结果。
2. 定义 Jira 只读字段最小集与错误出口。
3. 定义 GitLab artifact 标准结构与链接生成规则。
4. 定义 Feishu 目标定位、模板与锚点解析结构。
5. 定义 repo workspace 只读输出与搜索摘要结构。
6. 定义跨 connector 的统一错误语义。

## 8. 测试与验收

单元/契约测试：

- capability 接口职责分离
- Jira issue snapshot 字段完整性
- GitLab artifact 标准化一致性
- Feishu 目标解析结构
- repo 搜索与模块候选输出

集成测试：

- registry 能按 capability 正确选择 connector
- 同一项目在不同阶段获得一致的 connector 选择结果

文档一致性检查：

- 不把 preview 或 execute 混入只读能力
- 不允许 workflow 越层访问底层平台

验收标准：

- F04、F06 能直接消费标准化只读结果
- 平台细节被稳定封装在 Infrastructure 内

## 9. 与其他 Feature 的交接点

- 向 F04 提供 Jira、repo、GitLab 的分析输入。
- 向 F06 提供 Feishu 目标解析和 GitLab artifact 标准表达。
- 向 F07 提供 `inspect connectors` 与只读检查的基础能力。
