# F06 补录写回与幂等执行

## 1. 目标与用户价值

本 feature 负责把外部补录、preview、审批绑定、真实写入、幂等去重、dry-run 和 reconcile 组织成一个完整的副作用执行闭环。

用户价值如下：

- 用户可以先预览写回内容，再批准，再执行真实写入。
- 相同副作用请求在重试、恢复和不同入口下都不会被重复执行。
- 外部写回失败或结果未知时，系统能保留现场并支持后续恢复。

## 2. 来源依据

### 2.1 已确认事实

- 所有外部写操作都必须遵守 `preview -> approve -> execute`。
- GitLab 产物和 verification 允许在外部完成后补录回流程。
- `dry-run` 允许读取和 preview，但不允许真实写入。

### 2.2 上游约束

- 技术方案将 preview / execute / 错误标准化归入 Infrastructure Layer。
- 实施计划任务六冻结了 preview 规则、审批绑定、execute 门禁、marker、幂等键、模式差异和 reconcile。
- side-effect ledger 顺序由 F02 冻结，不得改变。

### 2.3 本 Feature 实现决策

- 写入类 canonical draft 由 F04 提供，本 feature 只负责将其转成可执行闭环。
- preview 版本与审批强绑定，旧 preview 审批必须失效。
- 相同业务写入无论从 `run` 还是 `record` 入口触发，都共用同一幂等与去重规则。

## 3. 范围

### 3.1 本 Feature 要做什么

- 定义 GitLab 产物与 verification 的补录方式。
- 定义 Jira / Feishu preview 生成规则与稳定字段。
- 定义 preview 与 approval binding 的关系。
- 定义 execute 前置条件、marker、`idempotency_key` 和 dedupe 规则。
- 定义 `dry-run`、`--non-interactive` 和 reconcile 行为。

### 3.2 本 Feature 明确不做什么

- 不重新定义八阶段主流程。
- 不承担 CLI 命令树设计。
- 不生成分析类业务结论。
- 不替代 F02 的 ledger 持久化规则。

核心对象：

- `ApprovalRecord`
- `SideEffectLedgerEntry`
- `StructuredError`

关键状态：

- `approved_pending_write`
- `executing_side_effect`
- `outcome_unknown`

关键字段：

- `preview_ref`
- `preview_hash`
- `marker`
- `target_ref`
- `idempotency_key`

能力边界：

- preview 的正式 owner 是 Infrastructure。
- approval 真值属于 workflow。
- ledger 顺序与终态属于 storage / workflow 协同规则。

## 6. 实现设计

### 6.1 主要流程

主要流程如下：

1. 用户补录 GitLab artifact 或 verification。
2. workflow 判断补录输入是否满足推进条件。
3. Infrastructure 根据 canonical draft 生成 preview。
4. 用户审批当前 preview。
5. execute 前执行 preview 一致性校验、幂等检查和 ledger / checkpoint 准备。
6. 真实请求发出后记录结果；若结果未知则转入 reconcile。

### 6.2 输入与输出

输入：

- GitLab commit / branch / MR 引用
- verification 摘要
- canonical draft
- preview 确认值
- `ProjectProfile` 中的写回目标配置

输出：

- preview 对象
- execute 执行结果
- ledger 相关线索
- reconcile 结果

### 6.3 关键约束

- 旧 preview 不能继续执行。
- execute 必须校验审批通过、preview 一致、目标去重通过、ledger 准备完成、checkpoint 已落盘。
- `dry-run` 禁止成功副作用记录。
- `outcome_unknown` 场景必须先 reconcile，再决定是否重试。
- 相同业务请求在不同入口下必须共用去重规则。

### 6.4 异常与失败分支

- 缺审批或 preview 不匹配：返回 `preview_version_stale` 或 `invalid_state`。
- 目标重复：阻止真实写入并返回重复写入结果。
- 外部写入明确失败：记录失败账本并保留上下文。
- 外部写入结果未知：进入 reconcile，不直接重放。

## 7. 开发任务拆解

1. 定义 artifact / verification 补录结构和最小字段。
2. 定义 preview 结构、稳定字段和允许变化字段。
3. 定义 preview 与 approval binding 规则。
4. 定义 execute 门禁与幂等检查流程。
5. 定义 `marker`、`target_ref`、`idempotency_key` 的统一生成口径。
6. 定义 `dry-run`、`--non-interactive` 和 reconcile 分支。

## 8. 测试与验收

单元/契约测试：

- preview 生成规则
- preview 与 approval binding
- execute 门禁
- marker / 幂等键 / 去重规则
- 模式差异
- reconcile 结果分类

集成测试：

- 外部补录后可生成 preview 并进入写回审批
- 恢复执行不会重复写入已成功副作用
- `outcome_unknown` 场景下先对账再决定是否重试

文档一致性检查：

- 不把 execute 逻辑提前到审批阶段
- 不让 `record` 绕过 preview 与 dedupe

验收标准：

- 写回链路可预览、可批准、可执行、可恢复
- 幂等与对账逻辑能阻断重复副作用

## 9. 与其他 Feature 的交接点

- 从 F04 获取 canonical draft。
- 与 F05 共同决定审批状态和 execute 入口。
- 向 F07 提供 `provide-artifact`、`provide-verification`、`preview-write`、`execute-write` 的行为基础。
- 向 F08 提供 Jira / Feishu 写回结果与恢复证据。
