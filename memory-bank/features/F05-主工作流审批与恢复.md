# F05 主工作流审批与恢复

## 1. 目标与用户价值

本 feature 负责把系统的八阶段主流程、审批门、回退、恢复和部分成功语义组织成一个一致的 workflow 内核。

用户价值如下：

- 用户能沿着固定主流程推进，而不是在多个入口之间失控跳转。
- 审批、退回和恢复执行具备一致语义。
- 外部补录、部分成功、过期产物和恢复行为都能被可预测地处理。

## 2. 来源依据

### 2.1 已确认事实

- `Workflow/Agent Layer` 是唯一业务状态 owner。
- v1 主流程固定为八个阶段。
- `approve` 不等于真实写入；写入仍需显式 execute。

### 2.2 上游约束

- 技术方案把阶段编排、审批状态、checkpoint、恢复和副作用账本归入 workflow。
- 实施计划任务五冻结了八阶段流程、审批门、`Execution` 输入矩阵、`stale` / `superseded`、恢复规则和 `partial_success`。
- `run_status` 仅为兼容字段，不得作为真源。

### 2.3 本 Feature 实现决策

- 八阶段主流程保持固定，不新增旁路阶段。
- workflow 只消费 F04 的结构化结果和 F02 的 durable state，不在内部重复做分析。
- `waiting_approval`、`waiting_external_input`、`partial_success` 等状态保留独立语义。

## 3. 范围

### 3.1 本 Feature 要做什么

- 定义八阶段主流程的进入条件、输入与输出。
- 定义审批门、审批结果语义和后续动作。
- 定义 `Execution` 阶段的补录输入矩阵。
- 定义 `stale`、`superseded`、`partial_success` 和恢复规则。
- 定义基于 checkpoint 的 resume 行为。

### 3.2 本 Feature 明确不做什么

- 不承担 CLI 命令表达。
- 不直接实现 connector 平台细节。
- 不直接生成分析产物或 preview。
- 不负责最终报告的验收矩阵。

## 4. 前置依赖与输出产物

前置依赖：

- F02 durable state
- F03 只读输入
- F04 分析产物

输出产物：

- 八阶段 workflow 定义
- 审批门与状态迁移规则
- `Execution` 补录输入矩阵
- 回退、过期、恢复、部分成功的状态语义

## 5. 涉及的核心对象、状态与能力边界

核心对象：

- `ExecutionContext`
- `ApprovalRecord`
- `CheckpointRecord`
- `StructuredError`

关键状态：

- `waiting_approval`
- `waiting_external_input`
- `approved_pending_write`
- `stale`
- `superseded`
- `partial_success`

能力边界：

- workflow 决定何时推进、等待、回退、恢复。
- storage 负责 durable 结果，不定义状态机。
- CLI 只触发动作，不决定状态真相。

## 6. 实现设计

### 6.1 主要流程

主流程固定为：

1. `Intake`
2. `Context Resolution`
3. `Requirement Synthesis`
4. `Code Localization`
5. `Fix Planning`
6. `Execution`
7. `Artifact Linking`
8. `Knowledge Recording`

阶段推进规则：

- 分析阶段先产出结构化结果，再进入审批或下一阶段。
- `Execution` 阶段负责等待 GitLab 产物与 verification 补录。
- 写入阶段先产出 preview，再进入审批与 execute。

### 6.2 输入与输出

输入：

- F04 产出的结构化分析结果
- F02 提供的 durable state 与 ledger 状态
- F06 提供的 preview / execute 执行结果

输出：

- 更新后的 `ExecutionContext`
- 审批记录
- 过期状态标记
- 可恢复 checkpoint

### 6.3 关键约束

- `approve` 与 execute 严格分离。
- 回退后旧 artifact 与旧审批必须失效，但历史记录保留。
- 恢复时不得静默重跑已批准分析阶段。
- 已成功副作用不得因恢复或 revise 被重放。
- `partial_success` 必须允许只重试失败部分。

### 6.4 异常与失败分支

- 错误阶段触发审批或写入：返回 `invalid_state`。
- `Execution` 输入不完整：保持 `waiting_external_input`。
- 恢复时发现未终态副作用：先进入 reconcile。
- 用户选择 `reject`：run 终止但保留审计与上下文。

## 7. 开发任务拆解

1. 定义阶段枚举、阶段输入输出和状态不变式。
2. 定义审批门与 `approve` / `reject` / `revise` 的状态行为。
3. 定义 `Execution` 的输入矩阵和合并策略。
4. 定义 `stale` 与 `superseded` 的过期规则。
5. 定义 resume 行为与 checkpoint 绑定关系。
6. 定义 `partial_success` 的最终结果表达与后续动作。

## 8. 测试与验收

单元/契约测试：

- 八阶段主流程完整性
- 审批分支流转
- `Execution` 输入矩阵
- `stale` / `superseded` 语义
- resume 规则
- `partial_success` 规则

集成测试：

- 标准主流程可从 Jira issue 一直推进到报告导出前
- revise 后旧结果不会继续参与后续步骤
- 中断恢复不会重复执行已成功副作用

文档一致性检查：

- 审批通过不等于执行写入
- `run_status` 只作为兼容字段

验收标准：

- workflow 具备唯一状态真源
- 回退、等待和恢复行为可预测且可追踪

## 9. 与其他 Feature 的交接点

- 从 F02 获取 checkpoint、ledger 和 durable state。
- 从 F04 获取分析产物和 canonical draft 输入。
- 与 F06 协作完成 preview / execute 状态推进。
- 向 F07 暴露 `run`、`approve`、`revise`、`resume` 等命令语义。
