# AGENTS.md

## 1. 文档目的

本文件定义当前仓库中所有 LLM agent 的统一协作规则，适用于以下类型的任务：

- 需求整理与补充
- 技术设计与方案收敛
- 实施计划拆解
- 代码实现与重构
- 测试验证
- 文档同步与知识沉淀

本文件不替代需求文档、技术方案或实施计划。它的职责是约束 agent 在本项目中如何读取信息、如何做决策、如何交付结果。

## 2. 项目背景

当前仓库用于定义和推进 `Bugfix Orchestrator Agent + Skills + CLI` 项目。该项目是一个面向本地开发场景的 `CLI-first` bugfix 编排系统，目标是在 Jira、GitLab、飞书云文档和本地代码仓库之间建立可配置、可审计、可恢复的协作闭环。

当前仓库阶段以文档先行为主，核心约束和边界已经记录在 `memory-bank` 目录中。Agent 的默认职责不是自由发挥产品方向，而是在既有文档基线之上推进一致、可验证、可追溯的工作。

## 3. 信息优先级

Agent 在本仓库中工作时，必须遵守以下信息优先级：

1. 用户当轮明确指令
2. `memory-bank/需求文档.md`
3. `memory-bank/技术方案.md`
4. `memory-bank/实施计划.md`
5. 本文件 `AGENTS.md`

如果高优先级信息与低优先级信息冲突，必须以高优先级为准。Agent 不得为了简化实现、补全文档或追求自动化，擅自改写上位文档已经明确的边界、职责或非目标。

## 4. Agent 工作模式

本项目采用“文档驱动、阶段感知、低风险自主、高风险升级”的工作模式。

### 4.1 文档驱动

- 收到任务后，先判断该任务对应的直接依据文档。
- 涉及需求边界时，优先回看 `需求文档`。
- 涉及分层职责、对象设计、状态机、模块边界时，优先回看 `技术方案`。
- 涉及执行顺序、步骤拆解、测试要求时，优先回看 `实施计划`。

### 4.2 阶段感知

Agent 必须先判断当前任务属于哪一类，再决定行动方式：

- 需求澄清
- 技术设计
- 实施拆解
- 代码实现
- 测试验证
- 文档同步

不得用实现细节反向覆盖需求定义，也不得用局部代码现状否定上位文档中的既定架构。

### 4.3 低风险自主推进

以下事项通常允许 agent 自主推进：

- 补充已有文档中的缺漏表述
- 修正明显术语不一致
- 在现有范围内补充分步说明
- 补充与当前改动直接相关的测试
- 小范围整理结构而不改变职责边界

### 4.4 高风险事项升级确认

遇到以下情况，必须先明确风险与影响，再请求用户确认：

- 新增或放宽 v1 范围
- 变更分层职责
- 变更核心对象、状态语义、审批门或主流程阶段
- 引入真实外部副作用
- 修改脱敏、持久化或安全策略
- 发现上位文档之间存在无法自行消解的冲突

## 5. LLM 开发规则

### 5.1 先理解，再改动

- 在修改任何文档、代码或配置之前，先阅读直接相关文件。
- 不得脱离仓库现状生成“通用最佳实践”式内容。
- 无法确认的信息，只能做最小必要假设，并明确标注为假设。

### 5.2 先对齐层次，再行动

- 需求问题回到需求文档处理。
- 架构问题回到技术方案处理。
- 执行顺序和验收要求回到实施计划处理。
- 不得跳过上游层次，直接在下游产物中私自重定义约束。

### 5.3 小步交付

- 默认一次只推进一个明确步骤、一个最小文档闭环或一个最小可验证改动。
- 完成一个步骤后应立即补对应验证。
- 不要堆积大量未验证改动后再一次性说明“后续再补测试”。

### 5.4 文档与实现同步

- 如果改动影响术语、流程、目录、对象、状态或职责，必须同步检查相关文档是否需要更新。
- 如果本轮不适合同步更新全部文档，必须明确指出已知不一致点。
- 任何影响术语、阶段名称、核心对象、状态语义的改动，优先保持与 `memory-bank` 文档一致。

### 5.5 禁止擅自扩 scope

除非用户明确要求且上位文档已同步更新，否则不得擅自加入以下能力：

- 自动修改代码
- 自动执行测试
- 自动创建 commit、branch 或 MR
- 多用户调度
- 自然语言对话式入口
- 超出 v1 定义的自治能力

### 5.6 优先结构化输出

Agent 输出应尽量清晰、可审阅、可执行，优先采用结构化表达，包括但不限于：

- 范围结论
- 设计决策
- 实施步骤
- 测试说明
- 风险清单
- 待确认事项

### 5.7 保持可追溯

- 关键结论应尽量能追溯到需求文档、技术方案、实施计划或用户明确指令。
- 不得将猜测、经验判断或默认偏好伪装成既定事实。

### 5.8 测试是交付的一部分

- 只要任务涉及变更，就应提供相应验证。
- 验证可以是测试、文档一致性检查、结构检查、命令结果核对或最小场景演练。
- 未验证的内容不得表述为“已完成且可用”。

## 6. 输出与交付要求

### 6.1 文档任务

- 优先直接更新目标文档，而不是只在对话中给建议。
- 说明本次新增、删改或重组的内容。
- 说明修改依据来自哪份文档或哪条用户指令。

### 6.2 实现任务

交付时至少应说明：

- 改了什么
- 为什么改
- 依据什么改
- 做了什么验证
- 还剩哪些风险、假设或待确认项

### 6.3 分析任务

分析类输出应主动区分：

- 已确认事实
- 文档已有约束
- Agent 的合理假设
- 需要用户确认的问题

### 6.4 多文件影响

如果修改会影响多个文档、模块或阶段，需要说明：

- 哪些地方已经同步
- 哪些地方仍待同步
- 是否存在暂时接受的不一致

### 6.5 验证记录

验证结果应尽量具体，至少说明：

- 验证对象
- 触发方式
- 预期结果
- 实际结果

## 7. 禁止事项

Agent 在本项目中不得进行以下行为：

- 绕过 `需求文档 -> 技术方案 -> 实施计划` 的约束链路自定方向
- 将 v1 非目标偷偷写入设计、计划或实现
- 混淆 `CLI Layer`、`Workflow/Agent Layer`、`Skill Layer`、`Infrastructure Layer`、`Renderers` 的职责
- 在没有依据的情况下虚构外部系统能力、字段、接口或仓库现状
- 将未验证的结果表述为已完成事实
- 以“更自动化”为理由削弱显式配置、人工确认、预览后执行、可恢复、可审计、脱敏落盘等核心原则

## 8. 必须升级确认的情况

出现以下情况时，必须暂停直接推进，先与用户对齐：

- 需要改变产品范围或非目标
- 需要改变系统分层或核心对象
- 需要新增、删除或重定义主流程阶段、审批门、状态语义
- 需要引入真实外部写操作或改变副作用策略
- 需要修改安全、脱敏、持久化或恢复机制
- 需求文档、技术方案、实施计划之间存在实质冲突，且无法通过局部解释解决
- 用户最新要求与现有文档明显冲突，且需要决定是临时例外还是更新基线

## 9. 默认执行准则

如果用户没有特别说明，Agent 应默认遵守以下准则：

- 优先维护文档一致性和边界清晰度
- 优先推进最小闭环，而不是扩展额外能力
- 优先做可验证改动，而不是一次性大改
- 优先显式说明假设和风险，而不是隐含处理
- 优先把结果落到仓库文件中，而不是停留在对话里

本文件的目标不是限制 agent 的效率，而是确保所有推进都与当前项目的真实阶段、既有方案和交付边界保持一致。
