## 4. 真实使用中的问题

### 4.1 `config_version` 容易踩坑

用户最初给出的是 `1.0.0`，但系统要求必须使用 `YYYY-MM-DD`。

真实结果：

- `1.0.0` 会被判定为 `invalid_version`
- 改成 `2026-03-23` 后才通过

### 4.2 Jira `issue_type_id` 不友好

用户只能提供 Jira 类型名称“需求”，拿不到内部 `issue_type_id`。  
本轮为了继续试用，只能先用占位值补配置。

结论：

- 当前 Jira 配置对首次用户门槛较高

## 5. 主流程与子工作流测试结果

### 5.1 在当前真实场景下不可用

已实际执行：

```bash
bugfix-orchestrator run start --project feat_owner --issue MPR-30577 --json
bugfix-orchestrator run brief --project feat_owner --issue MPR-30577 --json
bugfix-orchestrator record jira --project feat_owner --issue MPR-30577 --json
```

共同结果：

- 都返回 `validation_error`
- 错误信息均指向：`Jira issue MPR-30577 does not exist or is not visible to the current project configuration`

结论：

- 当前实现默认假设“从 bug Jira 启动”
- 在“只有需求 Jira、没有 bug Jira”的真实场景下，`run start`、`run brief`、`record jira` 都无法使用

### 5.2 可继续，但需要额外手工输入

已实际执行：

```bash
bugfix-orchestrator record feishu --project feat_owner --issue MPR-30577 --json
```

实际结果：

- 命令没有要求必须先有 bug Jira
- 但要求必须补齐以下 4 项：
  - `problem`
  - `root_cause`
  - `fix_summary`
  - `verification_summary`

结论：

- `record feishu` 更适合当前真实场景
- 但仍依赖用户手工提供结构化摘要

## 6. 本轮结论

- 当前项目的配置维护链路是可用的
- `inspect config` 适合作为首次用户的主入口
- 当前主流程明显偏向 bug 场景，而不适合“只有需求 Jira”的真实开发场景
- `record feishu` 是当前最接近可继续落地的入口

## 7. 后续建议

如果继续按真实场景测试，下一步最有价值的是补齐这 4 个字段：

- `problem`
- `root_cause`
- `fix_summary`
- `verification_summary`

然后继续验证：

- `record feishu`
- `preview-write`
- `approve`
- `execute-write`
