# v2 验收结论

## 1. v2 可用功能

当前 v2 可以当作一个本地 `CLI-first` bugfix 编排与记录工具来使用，并且比 v1 多了“可真实交付给使用者”的主流程、子工作流和导出能力。

可直接使用的能力：

- `bind project|jira|requirements|gitlab|feishu|repo`
  - 维护项目画像配置
- `inspect config|graph|connectors`
  - 检查配置完整性、关系图和 connector 状态
- `run start`
  - 根据 `issue_key` 真实读取 Jira bug，并依次推进 `Intake`、`Context Resolution`、`Requirement Synthesis`、`Code Localization`、`Fix Planning`
- `run brief`
  - 创建 `brief_only` 子工作流 run，并生成真实 `Requirement Brief`
- `run status`
  - 查看当前 run 状态、等待原因、可执行动作
- `run resume`
  - 从最近 checkpoint 恢复，并在可继续时自动推进到下一步
- `run approve|reject|revise`
  - 记录审批、拒绝、回退
- `run provide-artifact`
  - 补录 GitLab 产物
- `run provide-verification`
  - 补录验证结果
- `run bind-branch`
  - 显式绑定 bug Jira 与开发分支
- `run ensure-subtask`
  - 生成或确认 Jira 开发子任务 preview
- `run provide-fix-commit`
  - 显式补录 fix commit 归属
- `run preview-write`
  - 为 `Artifact Linking` 或 `Knowledge Recording` 生成真实 preview draft、渲染结果和结构化 payload
- `run execute-write`
  - 基于已批准 preview 执行真实 connector/stub connector 契约，并记录 result artifact
- `run report`
  - 生成统一 `Bugfix Report`
- `record jira`
  - 创建仅 Jira 回写子工作流 run，并支持显式补录 branch、artifact、verification、commit、subtask
- `record feishu`
  - 创建仅飞书记录子工作流 run，并支持最小上下文补录、preview、approve、execute

## 2. v2 最简操作方式

### 2.1 先绑定项目配置

```bash
bugfix-orchestrator bind project --project <project_id> --file <project.json>
bugfix-orchestrator bind jira --project <project_id> --file <jira.json>
bugfix-orchestrator bind requirements --project <project_id> --file <requirements.json>
bugfix-orchestrator bind gitlab --project <project_id> --file <gitlab.json>
bugfix-orchestrator bind feishu --project <project_id> --file <feishu.json>
bugfix-orchestrator bind repo --project <project_id> --file <repo.json>
```

与 v1 相比，`jira.json` 和 `gitlab.json` 至少还应补齐：

- Jira 子任务配置
- Jira branch 绑定规则
- Jira commit 绑定规则
- GitLab branch binding 输入模式

### 2.2 检查配置

```bash
bugfix-orchestrator inspect config --project <project_id>
bugfix-orchestrator inspect graph --project <project_id>
bugfix-orchestrator inspect connectors --project <project_id>
```

### 2.3 走标准主流程

```bash
bugfix-orchestrator run start --project <project_id> --issue <JIRA-123>
bugfix-orchestrator run status --run <run_id>
```

当流程在审批门或等待外部输入处停住时，可继续使用：

```bash
bugfix-orchestrator run approve --run <run_id> --stage "Requirement Synthesis" --preview-ref <preview_ref>
bugfix-orchestrator run revise --run <run_id> --rollback-to "Requirement Synthesis"
bugfix-orchestrator run resume --run <run_id>
```

### 2.4 生成并导出 Requirement Brief

```bash
bugfix-orchestrator run brief --project <project_id> --issue <JIRA-123>
bugfix-orchestrator run brief --project <project_id> --issue <JIRA-123> --output ./requirement-brief.md
```

### 2.5 走 Jira 写回子工作流

```bash
bugfix-orchestrator record jira --project <project_id> --issue <JIRA-123>
bugfix-orchestrator run bind-branch --run <run_id> --branch <branch_name>
bugfix-orchestrator run provide-artifact --run <run_id> --file <artifact.json>
bugfix-orchestrator run provide-verification --run <run_id> --file <verification.json>
bugfix-orchestrator run ensure-subtask --run <run_id> --dry-run
bugfix-orchestrator run provide-fix-commit --run <run_id> --issue <JIRA-123> --commit <commit_sha>
bugfix-orchestrator run preview-write --run <run_id> --stage "Artifact Linking"
bugfix-orchestrator run approve --run <run_id> --stage "Artifact Linking" --preview-ref <preview_ref>
bugfix-orchestrator run execute-write --run <run_id> --stage "Artifact Linking" --preview-ref <preview_ref> --confirm <preview_hash>
```

### 2.6 走飞书记录子工作流

```bash
bugfix-orchestrator record feishu --project <project_id> --issue <JIRA-123>
bugfix-orchestrator run preview-write --run <run_id> --stage "Knowledge Recording"
bugfix-orchestrator run approve --run <run_id> --stage "Knowledge Recording" --preview-ref <preview_ref>
bugfix-orchestrator run execute-write --run <run_id> --stage "Knowledge Recording" --preview-ref <preview_ref> --confirm <preview_hash>
```

### 2.7 生成并导出 Bugfix Report

```bash
bugfix-orchestrator run report --run <run_id>
bugfix-orchestrator run report --run <run_id> --output ./bugfix-report.md
bugfix-orchestrator run report --run <run_id> --json
```

## 3. v2 比 v1 新增或补齐的关键能力

相较于 v1，v2 主要新增或补齐了以下真实用户能力：

- `run start` 不再只是初始化 run，而是会真实读取 Jira snapshot，并推进前五个分析阶段
- `run brief` 不再只是创建空的 `brief_only` run，而是会生成真实 `Requirement Brief`
- 新增 `run bind-branch`
- 新增 `run ensure-subtask`
- 新增 `run provide-fix-commit`
- `preview-write` 不再只是占位 preview ref，而是会输出真实 `rendered_preview`、`request_payload` 和 `request_payload_hash`
- `execute-write` 不再只是占位 result ref，而是会输出真实 result artifact、目标引用、目标版本、结果 URL 和请求 ID
- 新增 `run report`
- `record jira` / `record feishu` 不再只是创建空壳子工作流，而是具备最小可用补录、审批、预览、执行和恢复能力
- requirement mapping 歧义不再只能停在内部状态，可通过 CLI 人工补录并继续
- 缺失的 `feishu-recorder`、`approval-gate`、`connector-router`、`artifact-renderer` skill 已形成正式命名与统一导出

## 4. 当前推荐使用场景

v2 现在适合用于：

- 维护面向单项目的 bugfix 编排配置
- 从某个 Jira bug 启动一次标准分析 run，并在审批门停住或继续
- 为某个 bug 显式绑定开发分支、子任务和 fix commit
- 为 Jira / 飞书写回生成真实 preview，并在审批后执行受控写入
- 生成 `Requirement Brief` 和 `Bugfix Report`
- 在 requirement mapping 歧义、外部输入缺失或流程中断后，从最近 checkpoint 继续
- 作为本地开发场景下的可审计、可恢复、可追溯 CLI 编排工具使用
