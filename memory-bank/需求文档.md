# Bugfix Orchestrator Agent + Skills + CLI 需求文档

## 1. 文档目的

本文档定义一个面向本地开发场景的 `CLI-first` Bugfix Orchestrator Agent 的产品需求。该系统用于打通 `Jira`、`GitLab`、`飞书云文档`、本地代码仓库之间的协作闭环，帮助用户通过 CLI 命令完成从 bug 获取、需求关联、代码定位、处理方案确认、外部系统回写到知识沉淀的全过程。

本文档仅描述需求，不包含实现方案、代码结构或开发任务拆解。

## 2. 背景与问题

当前 bug 修复流程通常分散在多个系统中：

- Bug 在 Jira 中流转
- 代码和提交在 GitLab 中管理
- 需求和复盘往往散落在飞书文档或本地文档中
- 开发者需要自行维护 bug、需求、项目、代码之间的关系
- 修复完成后，提交链接、问题原因、解决方案经常没有被系统化回写和沉淀

这导致以下问题：

- Bug 与需求关系不清晰
- 修复上下文难以追溯
- 提交与 Jira 关联依赖人工维护
- 经验无法沉淀为组织知识
- 不同项目的接入方式不统一，复用成本高

因此需要一个可配置、可审计、可逐步确认执行的 agent 系统，把这些环节串成一条稳定闭环。

## 3. 产品目标

系统需要实现以下核心目标：

- 让用户可以通过 CLI 命令发起一个 bugfix 流程
- 自动从 Jira 获取 bug 信息并解析关联需求
- 根据项目配置定位到正确的代码仓库和模块
- 生成一份完整的处理需求文档，供用户确认
- 在用户确认后推进后续分析与回写流程
- 在用户提供 GitLab 提交、分支或 MR 产物后，将对应链接回写到 Jira
- 在飞书云文档中记录该 bugfix 所属需求、问题描述、根因、解决方案和验证结果
- 将这套能力封装为多个 skill，供 agent 编排调用
- 提供 CLI 入口，支持通过配置命令绑定项目、需求、代码、Jira 之间的关系

### v1 最小闭环定义

v1 的最小闭环定义如下：

- 系统必须能从 Jira issue 出发，生成一份 `Requirement Brief`
- 用户确认后，系统必须能继续完成代码定位建议、修复计划生成、Jira 回写预览、飞书记录预览和 `Bugfix Report` 导出
- v1 不要求系统必须自动修改代码、自动执行测试、自动创建 commit、自动创建 MR
- v1 允许用户在外部完成代码修复后，将已有的 GitLab 产物重新带回流程中做关联与沉淀
- “Execution” 在 v1 中表示“推进修复相关动作的编排阶段”，而不是“系统必须自主改代码”

## 4. 非目标

v1 不包含以下目标：

- 不做服务端多用户协同调度
- 不做全自动无人值守修复
- 不承诺自动修改代码、自动跑测试、自动生成 commit
- 不做高自治知识图谱推理
- 不一次性支持所有协同平台
- 不负责复杂凭证托管平台建设
- 不替代 Jira、GitLab、飞书原有权限体系
- 不强制自动开 MR 或自动合并代码
- 不将自然语言对话入口作为 v1 必交付能力

## 5. 目标用户

主要用户包括：

- 开发工程师
- 技术负责人
- 需求/项目协同人员
- 使用 Codex 或类似 coding agent 的高级用户

用户特征：

- 熟悉本地命令行工作流
- 需要在多个协同系统之间切换
- 希望保留人工确认权，而不是完全交给 agent 自动执行

## 6. 使用场景

### 场景 1：标准 bug 修复闭环

用户输入一个 Jira bug 编号，agent 自动拉取 bug，识别对应需求和项目，生成一份执行简报。用户确认后，系统继续产出代码定位建议、修复计划、Jira 回写预览、飞书记录预览。用户在本地或外部完成修复后，再将 GitLab 产物带回系统，完成 Jira 关联与飞书沉淀。

### 场景 2：仅生成需求文档

用户输入一个 Jira bug 编号，但暂时不希望执行修复，只需要系统生成与该 bug 相关的需求/处理文档，供人工评审后再决定下一步。

### 场景 3：补录关联信息

某些项目尚未建立完整绑定关系，用户通过 CLI 命令补充 Jira 项目、GitLab 仓库、需求来源、飞书文档模板等配置，作为后续自动化运行的前提。

### 场景 4：单独补写记录

代码已经修复完，但 Jira 或飞书记录尚未更新。用户可单独触发“回写 Jira”或“写入飞书复盘记录”。

### 场景 5：需求映射失败后人工补录

系统从 Jira bug 中无法唯一识别所属需求时，应提示用户补录需求映射规则或手工指定目标需求，补录完成后允许从最近检查点恢复执行，而不是从头重新开始。

### 场景 6：审批退回与重新生成

用户在 `Requirement Brief` 或修复计划审批阶段选择 `revise`，系统应允许回退到指定上游阶段，重新生成结果，并将该阶段之后的旧产物标记为过期。

### 场景 7：外部写回失败但保留现场

系统在 Jira 或飞书写入阶段发生权限错误、网络错误或字段冲突时，应保留当前执行上下文、待写入 payload 和错误信息，供用户修正后重试。

### 场景 8：仅执行 dry-run 预览

用户希望先验证 Jira 回写与飞书文档写入内容是否正确，但不希望发生真实外部写操作。系统应仅输出预览结果和结构化 payload，不产生外部副作用。

### 场景 9：流程中断后恢复执行

系统在某阶段执行后中断退出，用户再次进入 CLI 时，应可从最近一个已持久化检查点恢复，并避免重复执行已经成功完成的外部写操作。

## 7. 产品形态

产品采用 `本地 CLI 为主、agent 编排为核心、skill 作为能力封装` 的形态。

说明：

- v1 的唯一必选入口是 CLI
- 自然语言对话入口属于后续扩展方向，不作为 v1 验收范围

系统包含四个逻辑层：

- `CLI 层`
  负责命令入口、配置管理、审批交互、结果展示
- `Workflow/Agent 层`
  负责按阶段编排 bugfix 过程
- `Skill 层`
  负责封装可复用能力
- `Connector 层`
  负责连接 Jira、GitLab、飞书以及未来可能的 MCP 接入

## 8. 核心原则

系统必须遵循以下原则：

- 显式配置优先，避免高风险自动推断
- 关键节点必须允许人工确认
- 所有外部写操作默认先预览、再执行
- 每个阶段都需要产出结构化结果
- 所有行为应可追踪、可审计、可恢复
- skill 之间不直接耦合，统一通过上下文传递状态
- connector 应隐藏接入细节，上层不关心底层是 API 还是 MCP

## 9. 项目关系绑定需求

系统必须支持维护一份项目画像配置，作为唯一可信来源。该配置至少应描述：

- 当前项目标识
- 配置版本
- Jira 实例与项目映射
- GitLab 仓库与分支规则
- 飞书文档空间与模板
- 需求来源位置
- 本地仓库路径与模块映射
- 流程执行策略与审批门禁
- 凭证引用方式
- 序列化与脱敏策略

关系绑定必须支持以下特点：

- 通过 CLI 显式维护
- 可按项目独立存储
- 可被 agent 运行时加载
- 缺失关键信息时必须提示补录，不能静默猜测
- 不同项目之间的凭证引用、模板配置和需求规则必须可隔离

### 项目画像配置最小字段要求

项目画像配置至少应包含以下字段：

- `project_id`
- `project_name`
- `config_version`
- `jira.base_url`
- `jira.project_key`
- `jira.issue_type_ids`
- `jira.requirement_link_rules`
- `jira.writeback_targets`
- `jira.credential_ref`
- `requirements.source_type`
- `requirements.source_ref`
- `gitlab.base_url`
- `gitlab.project_id`
- `gitlab.default_branch`
- `gitlab.branch_naming_rule`
- `gitlab.credential_ref`
- `feishu.space_id`
- `feishu.doc_id`
- `feishu.block_path_or_anchor`
- `feishu.template_id`
- `feishu.template_version`
- `feishu.credential_ref`
- `repo.local_path`
- `repo.module_rules`
- `approval_policy`
- `serialization_policy`
- `sensitivity_policy`

### 关联需求识别规则

系统必须支持显式定义从 Jira bug 到需求对象的识别规则，至少包含以下能力：

- 支持多种来源：`issue_link`、`custom_field`、`label`、`text_pattern`、`manual`
- 支持优先级排序
- 支持无法识别时的回退动作：阻塞、人工指定、降级为“未绑定需求”
- 支持多个候选需求时的人工确认
- 支持人工覆盖自动识别结果

### 未绑定需求处理策略

v1 对“未绑定需求”的默认处理策略如下：

- 系统允许在需求未成功绑定时继续生成 `Requirement Brief`、代码定位建议和修复计划
- 当需求未绑定时，必须显式标记 `requirement_binding_status = unresolved`
- 当需求未绑定时，`Requirement Brief`、`ExecutionContext`、`Bugfix Report` 中必须保留 `binding_reason`
- 飞书记录在需求未绑定时允许写入，但必须将“所属需求”明确标记为“未绑定需求”
- 若项目配置要求需求绑定为强约束，则系统必须在进入外部写回阶段前阻塞并提示人工补录

## 10. Skill 需求

系统需要将能力封装为多个 skill，至少包含以下领域 skill：

- `jira-intake`
  获取 bug 基础信息、状态、描述和关联线索
- `project-context`
  将 bug 与项目、仓库、模块、需求来源建立关联
- `requirement-summarizer`
  汇总 bug 与需求上下文，生成处理文档
- `code-locator`
  根据上下文推断候选代码位置和影响模块
- `fix-planner`
  生成修复建议、影响范围和验证建议
- `gitlab-linker`
  负责 GitLab 产物与 Jira 的关联和回写
- `feishu-recorder`
  负责飞书云文档中的 bugfix 沉淀

还需要以下通用 skill：

- `config-loader`
- `approval-gate`
- `connector-router`
- `artifact-renderer`
- `report-writer`

所有 skill 必须满足：

- 输入输出清晰
- 错误语义清晰
- 可独立调用
- 可被编排
- 不直接持久化业务状态

补充约束：

- Workflow/Agent 层是唯一业务状态持有者
- CLI 层只负责交互、展示和持久化序列化结果
- skill 只返回结构化结果，不直接持久化 run state
- connector 层必须负责外部系统能力检查、目标解析、写入预览和写入执行

## 11. 工作流需求

系统必须支持一条固定的 bugfix 主流程，至少分为以下阶段：

1. `Intake`
   输入 Jira issue key，拉取 bug 数据
2. `Context Resolution`
   解析该 bug 所属项目、代码仓、需求来源
3. `Requirement Synthesis`
   生成前置处理需求文档或执行简报
4. `Code Localization`
   提供候选代码位置与根因假设
5. `Fix Planning`
   给出修复建议和验证建议
6. `Execution`
   在确认后推进修复相关动作，例如调用外部 coding agent、记录人工修复产物或等待用户提供修复结果
7. `Artifact Linking`
   将 GitLab 产物回写到 Jira
8. `Knowledge Recording`
   将处理结果记录到飞书文档

工作流要求：

- 每个阶段都应可暂停
- 每个阶段都应能输出结构化结果
- 每个阶段都应能被恢复执行
- 用户拒绝继续时，系统应保留中间结果
- 外部写操作前必须允许预览

### 状态机要求

系统必须区分“阶段状态”“审批决定”和“run 状态”，不得混用。

所有阶段必须遵循统一阶段状态机，最小状态集合如下：

- `not_started`
- `in_progress`
- `output_ready`
- `waiting_approval`
- `waiting_external_input`
- `completed`
- `failed`
- `stale`

审批决定必须独立记录，最小集合如下：

- `approve`
- `reject`
- `revise`

run 状态最小集合如下：

- `active`
- `paused`
- `cancelled`
- `completed`
- `partial_success`
- `failed`

状态机要求：

- 阶段输出生成后必须从 `in_progress` 进入 `output_ready`
- 需要审批的阶段必须从 `output_ready` 进入 `waiting_approval`
- 用户选择 `approve` 后，当前阶段进入 `completed`，随后进入下一个阶段
- 用户选择 `reject` 后，当前 run 进入 `cancelled`，但必须保留审计记录和上下文
- 用户选择 `revise` 后，允许回退到指定上游阶段，并将后续阶段状态标记为 `stale`
- 当阶段需要等待用户补充 GitLab 产物、验证结果或其他外部输入时，必须进入 `waiting_external_input`
- 非审批阶段在成功产出最终结果后必须进入 `completed`

### 检查点与恢复要求

系统必须在以下时机持久化检查点：

- 阶段输出生成后
- 审批结果写入后
- 任一外部写操作成功后
- 任一外部写操作失败后

恢复执行要求：

- 恢复必须基于最近一个已持久化检查点，而不能依赖内存态
- 已批准阶段默认不得静默重跑，除非用户显式选择 `revise`
- 已成功执行的外部写操作必须具备去重依据，避免恢复时重复写入
- 处于 `waiting_external_input` 的阶段恢复后，系统必须能够继续等待用户补充输入，而不是错误重放该阶段

## 12. 审批与人工确认需求

系统必须支持“关键节点人工确认”模式，至少包括以下门禁：

- `Requirement Synthesis` 后
  用户确认需求文档/执行简报是否正确
- `Fix Planning` 后
  用户确认是否进入修复执行
- `Artifact Linking` 前
  用户确认 Jira 写回内容
- `Knowledge Recording` 前
  用户确认飞书写入内容

系统需要支持以下确认结果：

- `approve`
- `reject`
- `revise`

当用户选择 `revise` 时，系统应允许回到前一阶段重新生成结果。

补充要求：

- `revise` 必须支持回退到任一指定上游阶段，而不仅限于前一阶段
- 回退后，该阶段之后的所有产物都必须标记为过期
- `reject` 表示终止当前 run，但保留上下文与审计记录
- 审批决定必须独立记录到审批历史中，而不能仅隐含在阶段状态里

## 13. 需求文档生成需求

在 `Requirement Synthesis` 阶段，系统必须生成一份结构化文档，至少包含：

- Bug 摘要
- 关联需求
- 已知上下文
- 修复目标
- 待确认事项

该文档必须满足：

- 可在 CLI 中直接展示
- 可导出为 Markdown
- 可作为后续执行的输入依据
- 可被用户确认后继续推进流程

### Requirement Brief 最小字段集

`Requirement Brief` 至少应包含以下字段：

- `issue_key`
- `project_id`
- `linked_requirement`
- `requirement_binding_status`
- `binding_reason`
- `known_context`
- `fix_goal`
- `pending_questions`
- `generated_at`
- `source_refs`

当 `requirement_binding_status = unresolved` 时，`linked_requirement` 允许为空，但必须提供 `binding_reason`。

## 14. CLI 需求

系统需提供清晰的 CLI 命令体系，至少分为四类：

- `bind`
  维护项目画像和系统关系
- `inspect`
  做配置检查、关系预览、接入体检
- `run`
  启动主流程
- `record`
  单独补写 Jira 或飞书记录

首期建议支持的命令能力包括：

- 绑定 Jira 信息
- 绑定 GitLab 仓库
- 绑定飞书目标位置
- 绑定本地仓库和模块映射
- 绑定需求来源
- 检查配置完整性
- 预览当前项目关系图
- 启动 bugfix 流程
- 单独回写 Jira 链接
- 单独生成飞书记录

CLI 必须支持：

- 交互式输出
- JSON 输出
- dry-run 模式
- 明确错误码
- 阶段结果的序列化保存
- 恢复执行能力

### 子工作流要求

除完整主流程外，CLI 还必须支持以下子工作流：

- 仅生成 `Requirement Brief`
- 仅回写 Jira
- 仅写入飞书记录

子工作流必须满足：

- 具有明确前置依赖
- 不允许绕过必要审批
- 必须使用与主流程一致的状态机和错误语义

#### 子工作流最小前置输入

- `仅生成 Requirement Brief`
  最小输入为：`project_id`、合法 `issue_key`、可用 Jira 读取权限
- `仅回写 Jira`
  最小输入为：`project_id`、`issue_key`、至少一个 GitLab 产物引用、目标写回类型、可渲染的写回 preview
- `仅写入飞书记录`
  最小输入为：`project_id`、飞书写入目标、问题现象、根因摘要、解决方案摘要、验证结果摘要

若用户未从既有 run/checkpoint 恢复，则系统必须允许对上述最小输入进行手工补录。

## 15. Jira 集成需求

系统需要支持 Jira 的以下能力：

- 根据 issue key 获取 bug 信息
- 获取 bug 的状态、描述、指派、标签、关联需求线索
- 支持识别 bug 与需求之间的映射关系
- 支持在完成修复后回写 GitLab 提交或 MR 链接
- 支持更新指定字段或评论内容
- 在写回前提供预览
- 支持记录写入结果与目标对象标识

### Jira 写回契约要求

Jira 写回至少应包含以下字段：

- `issue_key`
- `target_type`
- `target_field_id_or_comment_mode`
- `rendered_preview`
- `request_payload`
- `idempotency_key`
- `result_id`
- `target_ref`
- `target_version`
- `result_url`
- `updated_at`

- 不在生成文档中泄露 token

补充要求：

- 必须支持对 `ExecutionContext` 和 `Bugfix Report` 做 redaction

## 22. 非功能需求

### 易用性

- 新项目可通过 CLI 快速完成绑定
- 常见操作应有明确提示和预览
- 支持分阶段确认，降低误操作风险

### 可扩展性

- 后续可新增更多 connector
- 后续可接入 MCP
- 后续可新增更多 workflow 和 skill

### 可维护性

- 不同系统接入边界清晰
- skill 之间职责明确
- 配置结构稳定可演化

### 可审计性

- 关键节点输入输出可回溯
- 外部写操作有预览和结果记录
- 中间产物可保留

### 优先级

- `P0`：写前预览、错误码、上下文持久化、恢复执行、敏感信息不泄露
- `P1`：JSON 输出、配置体检、单独补写
- `P2`：更多 connector、MCP 扩展能力、自然语言对话入口

### 成功指标

- 首次项目绑定在 15 分钟内完成
- 从 issue key 到生成 `Requirement Brief` 的中位耗时小于 60 秒
- 标准主流程成功率高于 90%
- 恢复执行成功率高于 95%
- 所有外部写操作 100% 支持 dry-run 预览

成功指标口径说明：

- 主流程成功率仅统计进入主流程且未被用户主动 `reject` 的 run
- 恢复执行成功率仅统计存在已持久化检查点的 run
- 因外部系统不可用或权限变更导致的失败应单独统计，不直接计入产品逻辑失败

## 23. 验收标准

系统达到以下标准时，可认为满足 v1 需求：

### 验收场景 1：标准主流程

- 给定一个已完成项目绑定的项目和一个合法 Jira issue key
- 当用户执行主流程命令
- 则系统必须生成 `Requirement Brief`
- 且在用户 `approve` 后继续生成代码定位建议、修复计划、Jira 回写预览、飞书写入预览
- 且最终导出一份结构化 `Bugfix Report`

### 验收场景 2：审批分支

- 给定一个进入审批阶段的执行实例
- 当用户分别选择 `approve`、`reject`、`revise`
- 则系统必须：
- 在 `approve` 后进入下一阶段并返回成功状态
- 在 `reject` 后终止当前 run 并保留上下文
- 在 `revise` 后回退到指定上游阶段，并将后续产物标记为过期

### 验收场景 3：配置缺失补录

- 给定一个缺少需求映射规则或外部系统关键字段的项目配置
- 当用户运行流程
- 则系统必须明确指出缺失项
- 且在用户补录后允许从最近检查点恢复，而不是从头重新执行整个流程

### 验收场景 4：dry-run 与真实写入

- 给定一个已生成 Jira/飞书写入草稿的执行实例
- 当用户执行 dry-run
- 则系统必须输出待写入 preview 和结构化 payload
- 当用户执行真实写入
- 则实际写入内容除系统生成的时间戳、链接 ID 外，必须与 dry-run 预览一致

### 验收场景 5：中断恢复

- 给定一个在任意阶段中断的执行实例
- 当用户执行恢复命令
- 则系统必须从最近一个已持久化检查点恢复
- 且不得重复执行已成功完成的外部写操作
- 且已获取上下文不得丢失

### 验收场景 6：部分成功与重试去重

- 给定一个执行实例已经成功完成 Jira 写回，但飞书写入失败
- 当用户修正权限、目标位置或 payload 后再次执行恢复
- 则系统必须保留 Jira 已成功结果
- 且不得重复执行已成功的 Jira 写回
- 且必须只重试失败的飞书写入步骤
- 且最终报告中必须明确记录该 run 曾发生部分成功

### 验收场景 7：需求映射歧义处理

- 给定一个存在多个候选需求或需求未绑定的 Jira issue
- 当用户运行主流程
- 则系统必须明确展示候选项或未绑定原因
- 且允许用户人工指定、人工跳过或按项目策略阻塞
- 且后续产物中必须保留最终的绑定状态和处理原因

### 验收场景 8：子工作流独立验收

- 给定一个满足最小前置输入的请求
- 当用户分别执行“仅生成 Requirement Brief”“仅回写 Jira”“仅写入飞书记录”
- 则每个子工作流都必须在不依赖完整主流程的前提下独立完成
- 且必须遵循相同的审批、错误和结果记录语义

### 演示路径

v1 必须支持一条标准演示路径：

- `bind -> inspect -> run -> approve/revise -> dry-run writeback -> real writeback -> export report`
- 每一步都必须有明确 CLI 输出、退出码和可检查产物

## 24. v1 边界与后续演进

### v1 范围

- 本地 CLI 运行
- Codex skill 为主
- Jira + GitLab + 飞书三方接入
- 显式配置驱动
- 人工确认驱动
- 需求文档生成与回写闭环
- 可恢复执行与结构化报告导出
- 外部写操作 dry-run 与真实执行双模式

### v1 明确不包含

- 自动修改代码
- 自动执行测试
- 自动生成 commit
- 自动创建 MR
- 自然语言对话入口

### 后续可扩展方向

- 服务端常驻 agent
- 多项目/多用户协作
- 自动 MR 创建与变更摘要
- MCP 优先接入
- 更多协同平台支持
- 更强的自动发现与推理能力

## 25. 默认假设

本文档基于以下默认假设：

- 用户希望先要一个稳定、可解释的闭环，而非全自动 agent
- 关系绑定以显式配置为主
- 关键节点必须允许人工审核
- 首期重点是把 Jira、代码、GitLab、飞书沉淀串起来
- “先生成需求文档，再确认执行”是主链路的一部分
- CLI 是主要操作入口，agent 负责编排，skill 负责能力复用
- Workflow/Agent 层拥有唯一状态机，CLI 和 skill 不拥有独立业务状态
