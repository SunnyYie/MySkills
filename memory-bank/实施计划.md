## 2. 全局执行规则

### 2.1 基本原则

- 每次只推进一个最小能力闭环，不把多个未验证改动堆在一起。
- 必须先补测试或先写验证说明，再做实现。
- 每完成一个步骤就立即完成对应验证，验证未通过不得进入下一步。
- 所有关键结论都要能追溯到需求文档、技术方案或用户指令。
- 影响术语、阶段、状态、对象、目录或职责的改动，必须同步检查相关文档。

### 2.2 分层约束

- `CLI Layer` 只负责命令入口、参数校验、交互和展示，不持有业务状态。
- `Workflow/Agent Layer` 是唯一业务状态 owner，唯一负责阶段流转、审批、回退、恢复和 checkpoint。
- `Skill Layer` 只接受结构化输入并返回结构化输出，不直接做 I/O，不持久化 run state。
- `Infrastructure Layer` 是唯一外部系统与本地仓库访问入口，负责读取、preview、execute、错误标准化。
- `Renderers` 只负责 CLI、Markdown、JSON 渲染，不参与业务推理。

### 2.3 风险约束

- 所有外部写操作都必须遵守 `preview -> approve -> execute`。
- 所有恢复动作都必须基于已持久化 checkpoint，不能依赖内存态。
- 所有真实外部写入都必须有幂等依据、账本记录和对账规则。
- 遇到 `outcome_unknown` 或存在 `prepared`/`dispatched` 但无终态的副作用记录时，恢复入口必须先进入 reconcile，禁止直接重发外部写入。
- 敏感字段不得明文落盘，不得出现在普通日志和导出文档中。
- 未在需求文档和技术方案中授权的能力，不得以“顺手补齐”为理由加入实现。

## 3. 跨任务契约

下列接口与类型视为跨任务契约，任何任务不得私自改名、弱化语义或改变 owner：

- CLI 命令组：`bind`、`inspect`、`run`、`record`
- 主流程阶段：`Intake`、`Context Resolution`、`Requirement Synthesis`、`Code Localization`、`Fix Planning`、`Execution`、`Artifact Linking`、`Knowledge Recording`
- 阶段状态：`not_started`、`in_progress`、`output_ready`、`waiting_approval`、`approved_pending_write`、`executing_side_effect`、`waiting_external_input`、`completed`、`failed`、`stale`、`skipped`
- 审批决定：`approve`、`reject`、`revise`
- 审批状态：`none`、`pending`、`approved`、`rejected`、`revise_requested`、`superseded`
- run 状态：`run_lifecycle_status` 与 `run_outcome_status` 双轨
- 核心对象：`ProjectProfile`、`ExecutionContext`、`RequirementBrief`、`BugfixReport`、`ApprovalRecord`、`SideEffectLedgerEntry`、`CheckpointRecord`、`StructuredError`
- Skill 契约：显式输入 schema、结构化输出 `StageResult<T>`、统一错误对象、无状态、不得直接做 I/O
- Connector 能力模型：`ReaderCapability`、`TargetResolverCapability`、`PreviewWriterCapability`、`SideEffectExecutorCapability`、`ArtifactResolverCapability`、`RepoWorkspaceCapability`

## 4. 统一步骤模板

每个任务组都必须沿用以下 5 个固定步骤，不允许跳步：

1. 先阅读本任务直接依赖的上位文档段落，并记录本任务不能突破的边界。
2. 先补测试或先写验证用例说明，再做实现。
3. 只完成当前最小能力，不顺手扩展后续能力。
4. 运行本步骤对应验证，确认通过后再进入下一步。
5. 检查是否需要同步更新文档、术语或测试夹具。

每一步的测试描述必须固定包含以下四项：

- 验证对象
- 触发方式
- 预期结果
- 通过标准
